## 2. Server-side vs client-side call split (cross-cutting â€” READ FIRST)

`mcp-client.ts` is a **browser-only** module: it reads `VITE_MCP_URL` via `import.meta.env`, holds `adminToken` as JS-module state, and its `exchange()` relies on the browser sending the `cg_session` HttpOnly cookie via `credentials: 'include'`. On the server (SvelteKit `+page.server.ts` load), none of that applies: there is no module state bound to a request, and the cookie must be forwarded manually.

**Decision:** M3 uses **client-side fetching** only. Every page ships a `.svelte` with a top-level `$effect` that calls `mcpClient.get(...)` after mount. No `+page.server.ts` files in M3. SvelteKit form actions (`+page.server.ts` `export const actions`) are **also not used** for mutation â€” mutations happen via `mcpClient.post()` in the client on button click. This keeps the admin-JWT lifecycle in one place (the browser module) and avoids a second server-side exchange path.

Consequence: the `+layout.server.ts` guard already blocks unauthenticated browsers at the shell level (`:32-34`), so the page code can assume `locals.session` is present by the time it renders. On initial paint the page shows a loading skeleton; `$effect` fires the admin-API call; on success the data renders.

**Every page in M3 therefore has `+page.svelte` only â€” no `+page.server.ts`, no form actions.** This overrides story prose ("SSR load fetches...") in STORY-006-03 Â§1 D1 and STORY-006-04 Â§1.
