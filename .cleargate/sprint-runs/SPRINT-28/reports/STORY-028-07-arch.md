## 2. Plan Deviations (orchestrator-confirmed) — both acceptable

### 2.1 `tsconfig.json` explicit options

**Deviation:** `extends: ".svelte-kit/tsconfig.json"` replaced with explicit `compilerOptions` + `allowImportingTsExtensions: true` + `noEmit: true` + new `admin/src/sveltekit-virtual.d.ts` for virtual-module type declarations.

**Verdict:** Acceptable. No production runtime impact (typecheck only). The new `.d.ts` ambient file is the canonical pattern for declaring SvelteKit virtual modules.

### 2.2 `LOCAL_MOCK_OVERRIDES` importer-detection in mcp-client loader hook

**Deviation:** Loader hook in `tests/setup-node-test-hooks.mjs` adds `importerIsTestFile` detection via `context.parentURL` — redirects `mcp-client.ts` → `__mocks__/mcp-client.ts` ONLY for non-test importers (component files). This was not in the plan; it emerged from running `mcp-client.node.test.ts` (which tests the real module) against the same loader-hook chain.

**Why required:** A single loader hook cannot be both "transparent for the SUT test" and "redirecting for the consumer-of-SUT test" without an importer-side filter. The `.node.test.ts` / `.test.ts` suffix check is the cheapest discriminator available at hook-resolve time.

**Verdict:** Acceptable. This is a well-scoped exception (only `mcp-client.ts`) and is documented inline with a comment block. It does not generalize into a sprawling allowlist.

## 3. Production-source pollution (QA flag) — accept as known-debt, surface in -028-08

**Files affected:**
- `admin/src/lib/stores/toast.svelte.ts` (+35 LOC) — exports `__toastMethods__` mutable record + wraps `toastStore` methods with an override check.
- `admin/src/lib/utils/clipboard.ts` (+9 LOC) — exports `__clipboardOverride__` and short-circuits `copyToClipboard()` when set.

**Why Dev added them:** `mock.module()` cannot intercept static ESM imports of already-loaded modules in `node:test`. Components import `toastStore.success()` and `copyToClipboard()` statically at module-init; by the time a test runs `mock.module()`, the binding is already frozen. The `__overrides__` mutable-state pattern is the in-band workaround.

**Architectural assessment:**
- **Runtime cost in production:** zero. Both override fields start `undefined` and never get assigned outside tests. The hot path adds one truthy-check per method call.
- **Conceptual cost:** non-zero. Production source now leaks test-seam exports. A future reader unfamiliar with the constraint will see `__toastMethods__` and wonder if it's a public API.
- **Reversibility:** medium. The pattern can be unwound via DI refactor (extract `toastStore` from a factory that accepts a `methods` arg) but that is multi-file invasive and out of EPIC-028 scope.

**Verdict:** Accept for SPRINT-28. The story-level alternative was "carve Bucket E to CR-068" — that would have shipped two-runner state, which the user explicitly forbade on 2026-05-17. The `__overrides__` pattern is the lower-cost trade.

**Recommendation:** STORY-028-08 documents the pattern (CLAUDE.md addendum or admin/CLAUDE.md) AND files a tracked SPRINT-29 cleanup item — `CR-XXX: Eliminate __overrides__ test seams in admin/ production sources via DI refactor`. Don't try to fit it into STORY-028-08.

This matches QA's framing ("technical debt, not a defect") and the FLASHCARD-worthy lesson: *node:test `mock.module()` only intercepts dynamic imports; static ESM bindings need in-source seams or a build-time alias layer.*

**Reading:** stub-internal `__overrides__` is uniformly named (good). The two prod-source seams (`__toastMethods__`, `__clipboardOverride__`) are the *real* fragmentation — and they're already on the cleanup list from §3 above.

**Verdict:** Not a sustainability blocker. The cleanup CR in SPRINT-29 should fold both prod-source seams into one DI pattern. Stub-internal overrides stay as-is — they're well-bounded.
