## STORY-002-09 scope resolution

**Inspection facts (before deciding):**
- Four synthesis recipes exist and are wired into `wiki-build.ts` lines 103-106: `cleargate-cli/src/wiki/synthesis/{active-sprint,open-gates,product-state,roadmap}.ts`. Each is `compile(state: RawItem[]) ‚Üí string`.
- Each recipe has a comment: `M3 placeholder body ‚Äî STORY-002-09 (M4) refines the template.` ‚Äî the M3 author explicitly handed off to 002-09.
- **`open-gates.ts` line 9 filters by `String(status).includes('üî¥')`.** Our corpus uses textual statuses (`"Draft"`, `"Ready"`, `"Active"` ‚Äî verified across 15+ pending-sync files). The filter matches zero items today and will keep matching zero forever. **This is a real bug, not just template polish.**
- `active-sprint.ts` lists every sprint regardless of `start_date`/`completed_at`/`activated_at`; the sprint frontmatter has those fields (SPRINT-04 file lines 6-9) but the recipe ignores them.
- `product-state.ts` lists epics only, dropping bug/CR/proposal counts.
- `roadmap.ts` lists every sprint + every epic in flat lists ‚Äî no chronology, no archived-vs-pending split.

**Decision: option (a) ‚Äî extract templates + fix data-shape bugs.** Reasons:
1. The `üî¥` filter is broken on real data. Leaving it ships an `open-gates.md` that's permanently empty. Failing CI is better than silent wrong output, but neither is acceptable.
2. Vibe-coders should be able to retune wording without touching TS ‚Äî option (b)/(c)/(d) all leave the templates buried in code. Extraction is the lowest-cost separation of concerns.
3. The recipes already isolate template strings as `lines: string[]` arrays ‚Äî extraction is mostly mechanical.

**Scope:** four `cleargate-cli/templates/synthesis/{name}.md` files + minor refactor to recipes to load + interpolate them via a small helper, plus the open-gates filter fix.

**JSON-merge algorithm (`merge-settings.ts` spec ‚Äî be exact):**
```
input: existing (parsed JSON or null), addition (the verbatim hook config below)
output: merged JSON

if existing is null:
  return addition

result = deep-clone(existing)
result.hooks ??= {}

for each [eventName, eventArray] in addition.hooks:
  result.hooks[eventName] ??= []
  for each newEntry in eventArray:
    matchingIdx = result.hooks[eventName].findIndex(e => e.matcher === newEntry.matcher)
    if matchingIdx === -1:
      result.hooks[eventName].push(newEntry)
    else:
      // matcher exists ‚Äî merge inner hooks[] by concatenation,
      // de-duped by exact `command` string match
      existingInner = result.hooks[eventName][matchingIdx].hooks ?? []
      for each newInner in newEntry.hooks:
        if !existingInner.some(h => h.command === newInner.command):
          existingInner.push(newInner)
      result.hooks[eventName][matchingIdx].hooks = existingInner

return result
```
Preserves `SubagentStop` (different event), preserves user's other PostToolUse matchers, idempotent on re-run (de-dup catches the second add).

**CLAUDE.md injection regex spec (`inject-claude-md.ts`):**
- Block format: literal `<!-- CLEARGATE:START -->\n<content>\n<!-- CLEARGATE:END -->`.
- Detection regex: `/<!-- CLEARGATE:START -->[\s\S]*?<!-- CLEARGATE:END -->/`.
- If `existing === null`: return the block as the entire file content (with trailing newline).
- If `existing` matches detection regex: `existing.replace(regex, block)` ‚Äî re-runnable, no duplication.
- If `existing` does NOT match: `existing.trimEnd() + '\n\n' + block + '\n'` ‚Äî append, single blank-line separator.
- Block content sourced verbatim from `cleargate-planning/CLAUDE.md` (the canonical `<!-- CLEARGATE:START -->...<!-- CLEARGATE:END -->` section, lines 7-40 of that file at plan time).

## Cross-story risks

- **Template-load path differs dev vs published.** 002-09 reads templates via `import.meta.url` from `dist/templates/synthesis/` after publish. Dev runs (vitest) read from `cleargate-cli/templates/synthesis/`. The `templateDir` test seam handles this. Both stories must thread the seam consistently ‚Äî 002-05's `initHandler` ALSO needs to forward template resolution to its inner `runWikiBuild` call so the bootstrap pass finds templates.
- **`prebuild` script timing.** If 002-05 lands the script but 002-09 hasn't yet shipped templates, `npm run build` succeeds (script just copies whatever exists). If 002-09 lands templates but 002-05 hasn't shipped the script, templates are gitignored from cleargate-cli but exist at meta-root ‚Äî published package won't include them. **Order matters: 002-09 commits templates + tsup config; 002-05 commits prebuild script.** Both must be present for a working publish. QA verifies via `npm pack --dry-run` listing.
- **Hook write-loop defense (third layer).** Init writes the M1-verified hook with `case "$FILE" in *.cleargate/delivery/*)` filter. STORY-002-07 (M3) also filters in CLI. STORY-002-02 subagent (M2) also filters. All three converge on `.cleargate/delivery/**` ‚Äî verify in M4 dogfood by intentionally writing to `.cleargate/wiki/foo.md` and confirming the hook does NOT re-fire.
- **`cleargate-planning/CLAUDE.md` block content drift.** 002-05 copies the bounded block from this file at runtime. If a future PR edits the block in `cleargate-planning/CLAUDE.md`, init's behavior changes silently. Add a snapshot test in `init.test.ts` that asserts the extracted block contains specific known sentinels (`"Session-start orientation"`, `"Halt at gates"`).
- **Open-gates filter fix is observable.** After 002-09 lands, running `cleargate wiki build` at meta-root will produce a non-empty `open-gates.md`. The pre-existing `wiki/open-gates.md` (committed by M3 dogfood, if any) will diff non-trivially. QA must regenerate the meta-root wiki post-002-09 commit and amend that into the same commit (or a follow-on dogfood commit).
- **Bootstrap pass and `wiki ingest` PostToolUse hook self-trigger.** Init's bootstrap pass writes ~50 wiki pages. If init runs in a Claude Code session AND PostToolUse is wired AND the hook fires on writes under `.cleargate/wiki/**`, infinite loop. **Three defenses in place:** (a) hook `case` filter excludes `wiki/**` (only `delivery/*` matches); (b) ingest CLI step 2 explicitly skips `wiki/**`; (c) `wiki build` is invoked directly in-process by init, bypassing hook entirely. Verify all three in scenario 7.
