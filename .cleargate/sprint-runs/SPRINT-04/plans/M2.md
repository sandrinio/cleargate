### STORY-002-03 — `cleargate-wiki-query` (model: haiku)

**Files to create (byte-identical):**
- `/Users/ssuladze/Documents/Dev/ClearGate/cleargate-planning/.claude/agents/cleargate-wiki-query.md`
- `/Users/ssuladze/Documents/Dev/ClearGate/.claude/agents/cleargate-wiki-query.md`

**Frontmatter (verbatim):**
```yaml
---
name: cleargate-wiki-query
description: Use AT TRIAGE before drafting any new Proposal, Epic, or Story — surfaces prior work that may already cover the topic. Read-only by default. Reads .cleargate/wiki/index.md first, then targeted per-item pages, returns synthesized answer with [[ID]] citations. Append `--persist` (via the `cleargate wiki query` CLI wrapper) to file the answer back to wiki/topics/<slug>.md as a topic page (Karpathy compounding loop).
tools: Read, Write, Grep, Glob
model: haiku
---
```

**Workflow steps:**
1. First-line role prefix `role: cleargate-wiki-query`.
2. Receive natural-language query string + optional `persist: true|false` flag.
3. **Read `.cleargate/wiki/index.md` first** — single Read call, no Glob. This is the entry point.
4. **Targeted lookups:** based on index hits, Read the specific per-item pages (`wiki/epics/<id>.md`, `wiki/stories/<id>.md`, etc.). Bound to ≤10 page reads per query (cost cap).
5. **Synthesize answer** — 3-6 sentences max, every claim cites at least one `[[WORK-ITEM-ID]]` backlink (§10.5).
6. **Read-only mode (`persist: false`, default):** return synthesized answer to caller. Do not write any file.
7. **`persist: true` mode:** compute slug from query (kebab-case, ≤40 chars). Write `.cleargate/wiki/topics/<slug>.md` with frontmatter:
   ```yaml
   ---
   type: topic
   id: "<slug>"
   created_by: "cleargate-wiki-query"
   created_at: "<ISO 8601 UTC>"
   cites: ["[[ID1]]", "[[ID2]]", ...]
   ---
   ```
   Body = the synthesized answer prose with `[[ID]]` citations preserved inline. Then update `wiki/index.md` "Topics" section (append one row).

**Guardrails:**
- **Default is read-only.** Topic-page writes require explicit `persist: true` (PROPOSAL-002 Q9 — "no auto-persist").
- Cite at least one `[[ID]]` per claim sentence. Uncited assertions are a lint violation in topic-page mode.
- Never write outside `.cleargate/wiki/topics/` and `.cleargate/wiki/index.md`.
- Never modify per-item pages — those are owned by ingest.
- Page-read budget ≤10 per invocation. If query needs more, return "narrow your query" rather than scanning the corpus.

**"What you are NOT":**
- Not ingest — do not create or update per-item pages.
- Not lint — do not flag drift.
- Not a search engine — return one synthesized answer, not a ranked list.

**Test scenarios from Gherkin (story §2):**
1. **Triage finds prior work** — Given `wiki/index.md` lists `PROPOSAL-stripe-webhooks` archived as `LIN-987`, When user prompts about "Stripe webhook support", Then subagent surfaces the archived Proposal with `[[PROPOSAL-stripe-webhooks]]` citation. (M3 STORY-002-08 wiki-query CLI test will assert this end-to-end; for M2 the def must specify the citation requirement so the test can assert presence of the bracket-token.)

**Reuse:**
- §10.5 backlink syntax — paste lines 287-289 verbatim into the def.
- §10.1 directory layout — reference `wiki/topics/` line 219 to enforce write boundary.

**Gotchas:**
- `persist` is a flag from the CLI (M3 STORY-002-08 implements `cleargate wiki query <q> --persist`), not an argv parsed by the subagent. The subagent receives it as a structured input field.
- Topic-page slug collisions: if `wiki/topics/<slug>.md` already exists, **overwrite** (latest synthesis wins). Lint catches stale-citation drift later.
- `wiki/index.md` "Topics" section may not exist yet on first persist. Def must spec creating the section header `## Topics\n\n` before appending the row, to keep index.md compile-recipe deterministic.

### STORY-002-04 — `cleargate-wiki-lint` (model: sonnet)

**Files to create (byte-identical):**
- `/Users/ssuladze/Documents/Dev/ClearGate/cleargate-planning/.claude/agents/cleargate-wiki-lint.md`
- `/Users/ssuladze/Documents/Dev/ClearGate/.claude/agents/cleargate-wiki-lint.md`

**Frontmatter (verbatim):**
```yaml
---
name: cleargate-wiki-lint
description: Use BEFORE Gate 1 (Proposal approval) and Gate 3 (Push) to enforce wiki-vs-raw consistency. Default mode (enforcement) exits non-zero on any drift, naming the offending page; halts gate transitions. `--suggest` mode (advisory) exits 0 and prints candidate cross-refs the ingest pass missed (Karpathy discovery). Performance: O(n), one pass per page plus one index cross-check; no all-pairs comparison.
tools: Read, Grep, Glob, Bash
model: sonnet
---
```

**Workflow steps:**
1. First-line role prefix `role: cleargate-wiki-lint`.
2. Receive mode flag (`enforce` default, `suggest` opt-in) from CLI invocation.
3. **Load wiki state once:** Glob `.cleargate/wiki/{epics,stories,sprints,proposals,crs,bugs,topics}/*.md` + Read `wiki/index.md`. One pass.
4. **Per-page checks (run for every page):**
   - **(a) Orphan:** if frontmatter `raw_path` does not exist on disk, FLAG `orphan: <page-path> -> missing <raw_path>`.
   - **(b) `raw_path` ↔ `repo` mismatch (A1):** apply the §10.4 derivation (`cleargate-cli/` → cli, `mcp/` → mcp, `.cleargate/`/`cleargate-planning/` → planning). If derived ≠ stored, FLAG `repo-mismatch: <page-path> declares <stored> but raw_path implies <derived>`.
   - **(c) Stale `last_ingest_commit` (A2/§10.7):** run `git log -1 --format=%H -- <raw_path>`. If stored ≠ current, FLAG `stale-commit: <page-path> at <stored>, current <head>`.
   - **(d) Missing-ingest:** if raw file mtime is newer than wiki page mtime (proxy for missed hook), FLAG `missing-ingest: <raw_path> newer than <page-path>`.
5. **Single index cross-check pass (§10.5 backlinks):**
   - For every `parent: [[X]]` in any page, verify page X exists AND page X's `children:` list contains `[[<this-page>]]`. FLAG `broken-backlink: <child> -> <parent>` on miss.
6. **Topic-page check (§10.8 invalidated citations):**
   - For every `wiki/topics/*.md`, parse `cites:` list. For each cited `[[ID]]`, look up the per-item page and check status. If status is `cancelled` or page is missing, FLAG `invalidated-citation: <topic-page> cites <id> (<reason>)`.
7. **Exclusion-respect check:** verify NO wiki page exists for any raw file under `.cleargate/{knowledge,templates,sprint-runs,hook-log,wiki}/`. If found, FLAG `excluded-path-ingested: <page-path>` (this catches a misbehaving ingest).
8. **`enforce` mode:** if any FLAG exists, exit 1; print all flags to stdout, one per line, prefixed with category. Otherwise exit 0.
9. **`suggest` mode:** in addition to running checks 4-7 (but NOT exiting non-zero on flags), scan for **candidate cross-refs ingest missed**: per-item pages whose body mentions another work-item ID in plain text (not in `[[ ]]`) AND that target page exists — emit as `suggest: <page> mentions <id>, consider [[ ]] wrap`. Always exit 0 in suggest mode.

**Guardrails:**
- **O(n), no all-pairs.** Per-page checks are local. Index cross-check is one linear scan over collected `parent:` declarations. Topic-cite check is one linear scan. Total work = O(pages + edges), not O(pages²). (Sprint risk row 3 — flashcard-worthy on first run.)
- **No raw-file ingest.** Lint is read-only on raw files; it only diffs against git.
- **Pagination at 50 per bucket** (PROPOSAL-002 §2.3): if any bucket exceeds 50 entries in `wiki/index.md`, FLAG `pagination-needed: <bucket>` — does not auto-paginate.
- Never modify any wiki page; reporting only.
- Suggest mode is purely additive — never silently mutates anything in either mode.

**"What you are NOT":**
- Not ingest — do not write or fix pages.
- Not query — do not synthesize answers.
- Not a CI check — gate-blocking is local-only; CI integration is deferred (sprint constraint, line 96).

**Test scenarios from Gherkin (story §2):**
1. **Orphan detected** — Given a story with `parent_epic_ref` pointing to missing EPIC, When lint runs, Then exit code > 0 and diagnostic names the orphan. (Maps to check 5 broken-backlink + check 4a orphan.)
2. **Stale summary** — Given a wiki page's summary references removed blast-radius items, When lint runs, Then the page is flagged for rebuild. (Maps to check 4c stale-commit — the underlying raw changed, so the wiki summary is suspect.)

Both scenarios verified end-to-end in M3 STORY-002-08 (wiki-lint CLI test). For M2, the def must specify exit-code conventions and flag-line format (`<category>: <path> -> <detail>`) precisely so M3 tests can assert on stdout shape.

**Reuse:**
- §10.8 lint-checks list — paste lines 331-337 verbatim into the def, then expand each into a workflow step with concrete predicate.
- §10.7 idempotency rule (lines 313-320) for check 4c.
- §10.3 exclusions list (lines 244-247) for check 7.

**Gotchas:**
- Sonnet (not haiku) because cross-page consistency requires relating multiple pages — but the def must keep work O(n), not encourage exhaustive reasoning. Be explicit: "do not pairwise-compare all pages."
- The story §2 second scenario phrases it as "stale summary" which is fuzzy; A2 makes this concrete = stale-commit. The def must reframe the Gherkin in terms of the §10.8 check-name (`stale-commit`) so M3 tests align.

## Cross-story risks

- **§10 page-schema drift across the three defs.** All three subagents reference the §10.4 frontmatter shape: ingest writes it, query reads `cites:` topic frontmatter, lint validates every field. Any per-def divergence in field names = lint catches but burns a QA kickback round. **Mitigation:** every Developer pastes the literal §10.4 block (protocol lines 256-278) inline; QA runs `diff` of the three pasted blocks across the three subagent files and asserts identical field-set. Flashcard recorded (`#wiki #cost #subagent`).
- **Exclusion list (§10.3) lives in TWO defs.** Ingest enforces it on write (skip); lint enforces it on audit (don't-flag-as-orphan AND check 7 catches violations). Both must paste the same five-entry list (the four §10.3 entries PLUS `.cleargate/wiki/` for ingest's loop-prevention). If they diverge, ingest could create pages lint then flags as orphans.
- **Topic-page contract spans query (writer) and lint (validator).** Query writes `wiki/topics/*.md` with `cites: [[ID1]]`. Lint check 6 reads that field and validates each citation. Both defs must agree on the `cites:` array shape — list of strings each containing one `[[ID]]` token. QA cross-checks.
- **The dogfood mirror is silent.** Three new files in `.claude/agents/` activate immediately on next Claude Code session. If a dev edits the canonical copy and forgets the live mirror (or vice versa), the dogfood instance behaves differently from what `cleargate init` will install. Mandatory `diff` post-edit (flashcard 2026-04-19 #wiki #protocol #mirror).
- **Hook spec belongs to M4, not M2.** Ingest's def **mentions** the PostToolUse hook as a trigger but does NOT include the hook JSON. The verified hook config lives in M1 plan lines 70-86 and gets written by STORY-002-05 in M4. Ingest def stays trigger-agnostic (works for hook OR direct invocation) — verified by reading §10.9 (lines 343-348).
- **CLI commands referenced but not yet implemented.** Ingest's step 9 invokes `cleargate wiki ingest <file>` which M3 STORY-002-07 ships. Query's `--persist` is M3 STORY-002-08. Lint's exit codes are consumed by M3 STORY-002-08. M2 defs **must** specify the contract precisely; M3 implements to match. If M2 spec is loose, M3 has to re-derive.
