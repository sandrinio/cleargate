**`count_tokens.mjs`**:
```javascript
#!/usr/bin/env node
// Header: purpose, args, exit codes, fallback chain (see Resilience block below).
import { parseJsonl, normalizeRow, aggregate } from './lib/ledger-digest.mjs';

function main() {
  const [sprintId, ...flags] = process.argv.slice(2);
  if (!sprintId) { /* usage; exit 2 */ }
  const json = flags.includes('--json');
  const sprintDir = process.env.CLEARGATE_SPRINT_DIR
    ? path.resolve(process.env.CLEARGATE_SPRINT_DIR)
    : path.join(REPO_ROOT, '.cleargate', 'sprint-runs', sprintId);
  const ledgerFile = path.join(sprintDir, 'token-ledger.jsonl');
  if (!fs.existsSync(ledgerFile)) {
    process.stderr.write(`Error: token-ledger.jsonl not found at ${ledgerFile}\n`);
    process.exit(1);   // Hard error: missing file (per story R4).
  }
  const rows = parseJsonl(ledgerFile).map(normalizeRow);
  if (rows.length === 0) {
    // Empty-ledger soft path (per story §1.4 Open Q #1 — accept recommended).
    if (json) {
      process.stdout.write(JSON.stringify({ total: 0, by_agent: {}, by_work_item: {}, anomalies: [] }));
    } else {
      process.stdout.write(`Ledger empty for ${sprintId} (0 rows).\n`);
    }
    process.exit(0);
  }
  const digest = aggregate(rows);
  if (json) {
    process.stdout.write(JSON.stringify(digest));
  } else {
    // Human format matching CR-021 §3.2.5 example.
    process.stdout.write(formatDigest(digest, sprintId));
  }
  process.exit(0);
}
```

#### Gotchas surfaced from code inspection

1. **`token-ledger.jsonl` rows have BOTH `story_id` and `work_item_id`.** Verified `.cleargate/sprint-runs/SPRINT-17/token-ledger.jsonl:1` — `"story_id":"","work_item_id":"BUG-004"`. Story R-Risk #2 says "schema-tolerant parser — accept `work_item_id || work_item || null`". The empty-string `story_id` case is real and must collapse to null in the aggregator (otherwise the `by_work_item` map gets a `""` key bucket). Fix in `normalizeRow`: `row.work_item_id || row.work_item || row.story_id || null` AND treat empty string as null: use `(row.work_item_id || row.work_item || row.story_id || '').trim() || null`.

2. **Token ledger SubagentStop attribution Red is in-flight.** Sprint Plan §3 risk row 1: rows in SPRINT-17/SPRINT-18 may be attributed to the orchestrator session, not the subagent. Story §1.3 explicitly puts the Red OUT of scope: "this story only **reads** what the ledger contains; it does not fix the upstream attribution." The aggregator MUST NOT crash when `agent_type` is missing or set to a value outside `{architect, developer, qa, reporter}` — bucket unknown types into `by_agent.unknown` (already covered by `normalizeRow` fallback to `'unknown'`).

3. **`git log --stat sprint/S-NN ^main`** silently fails when the branch is absent (story §3.2 step 3 says "exec ... 2>/dev/null" + "silent fail if branch absent"). Use `child_process.execSync` with `{ stdio: ['ignore', 'pipe', 'ignore'] }` AND wrap in try/catch — `execSync` throws on non-zero exit. On catch, the section content becomes `Git log unavailable for sprint/S-NN (branch not found).` matching the story R4 "reduce that subsection to a one-liner" pattern.

4. **Sprint-plan filename is NOT deterministic.** The plan file at `.cleargate/delivery/pending-sync/SPRINT-18_Prepare_Close_Observe_Mechanics.md` has a slug after the sprint-id prefix. Search by glob: `pending-sync/<sprint-id>_*.md` THEN fall back to `archive/<sprint-id>_*.md`. If both match (unlikely — pending-sync is the source of truth pre-close), prefer pending-sync.

5. **§-heading regex is fragile.** Sprint Plans use `## 1. Consolidated Deliverables` / `## 2. Execution Strategy` / `## 5. Risks` style (verified `SPRINT-18_*.md:56,88,167`). Match `^## ${num}\. ` (literal-dot, single space) — be tolerant to either Unicode `.` or Word-Number-Period since templates evolve. Capture from heading line until the next `^## \d+\.` or EOF. Test scenario 4's fixture must use the exact heading shape.

6. **FLASHCARD date-window filter — Strategy-3 fallback is mandatory.** FLASHCARD entries are tagged `YYYY-MM-DD · #tag1 ...`. The story R1 reads "entries dated within `[sprint.start_date, sprint.end_date]`". For SPRINT-18 `start_date: 2026-05-02`, `end_date: 2026-05-15` — but FLASHCARD-2026-05-01 entries (the day before sprint start) ARE relevant if their commits fall in-window. Apply Strategy-3 fallback PER FLASHCARD `2026-05-01 #closeout #script #fallback`: when the date filter returns 0 rows AND the sprint is closed (state.json `sprint_status === 'Completed'`), fall through to `git log --grep "<sprint-id>"` and surface commit subjects under a `Would-be flashcard candidates (Strategy-3 fallback):` sub-header. Do NOT apply Strategy-3 mid-sprint (false positives during active execution); gate on `sprint_status`.

7. **Bundle-size budget headroom.** 80KB cap per R3. Risk: SPRINT-17 has 4 milestone plans (M1.md..M4.md) and a multi-K-line sprint-plan body. Mitigation: `buildMilestonePlans` emits the verbatim plan files (per story R1) but `buildSprintPlanSlices` is sliced to §1/§2/§5 only — verify the sliced length per section ≤ ~10KB before concat. If totals exceed 80KB at compose-time, the script SHOULD log a warning to stderr but still write the file (do NOT truncate — partial bundles are worse than oversized; let the human decide). Add `process.stderr.write('Warning: bundle exceeds 80KB target — consider trimming sprint-plan §2 in-place\n')` when `bundleBytes > 81920`.

8. **`os.tmpdir()` is the test fixture root, NOT the real sprint-runs dir.** All 6 scenarios use `mktemp -d` (matches `test_close_pipeline.sh:32-34`). Inside, scenario 4/5/6 must build a synthetic `pending-sync/SPRINT-fixture_*.md` AND set `CLEARGATE_PENDING_SYNC_DIR=<tmp>` so `prep_reporter_context.mjs` finds it. Without this env-override, the test either green-flickers (real pending-sync has a SPRINT-fixture file? no) or red-flickers when run on a developer's machine with stale state. **Add the env override now, document in the script header, and use it from the test.**

## Cross-story risks

A Developer working on STORY-025-01 in isolation might miss:

- **STORY-025-03 (Wave 2) consumes `prep_reporter_context.mjs` from `close_sprint.mjs` Step 3.5.** If STORY-025-01's CLI shape changes between Wave 1 and Wave 2 (e.g., adds a required flag, changes positional arg order), STORY-025-03's `invokeScript('prep_reporter_context.mjs', [sprintId], { ... })` call breaks. **Lock the CLI shape now: positional `<sprint-id>` only; no required flags.** Document the contract in the script header so STORY-025-03's M3 plan can lift it verbatim.

- **STORY-025-05 (Wave 3) updates `reporter.md` to reference `.reporter-context.md` as default input.** If STORY-025-01 changes the output filename (e.g., `.reporter-bundle.md`) post-Brief, STORY-025-05's M5 plan diverges. **Lock the filename now: `.reporter-context.md`** (matches CR-021 §3.2.4 + story R1 verbatim).

- **`count_tokens.mjs --json` schema is consumed by `prep_reporter_context.mjs` AND will be referenced in STORY-025-05's reporter.md Capability Surface table.** If the JSON shape changes, two downstream consumers break. **Lock the JSON keys: `total`, `by_agent`, `by_work_item`, `anomalies`** — exactly per story §2.1 Gherkin scenario 5. Treat this as a stable public API.

- **Token-ledger Red carry-over (Sprint Plan §3, story §1.3):** because attribution may collapse to orchestrator sessions, the `by_agent` breakdown for SPRINT-17 may show ~all tokens under `architect` (the orchestrator runs as architect-tier). The Reporter context bundle will still show real numbers — just attributed wrong. **Do NOT treat this as a test failure** — Gherkin scenario 4's "per-agent breakdown line for each of architect/developer/qa/reporter" assertion is satisfied if the line exists, even with zero on three of four agents. **The fixture for scenario 4 must seed all 4 agent-types explicitly** so the test passes deterministically regardless of the upstream Red.

## Open decisions for orchestrator

1. **Helper extraction — pre-emptive vs. conditional?** Story R5 says "extract into `lib/ledger-digest.mjs` if duplication exceeds 20 LOC". My recommendation above is to pre-emptively create the lib file. **Decision needed:** does the orchestrator/Architect want strict adherence to the ">20 LOC" gate (Developer first writes both scripts, measures duplication, then extracts) or the pre-emptive shape (lib file from line 1)? **Recommendation: pre-emptive.** Two consumers + a 60-LOC helper hits the threshold immediately; conditional extraction wastes a refactor pass.

2. **Test format — bash `.sh` (matches neighbours) vs. vitest under `cleargate-cli/test/scripts/`?** Story §3.1 mentions both options ("`.cleargate/scripts/test/test_*.sh` (NEW) ... or vitest equivalents under `cleargate-cli/test/scripts/`"). Fewer test-runtime dependencies. CR-023's `pool: 'forks'` vitest fix in Wave 1 is irrelevant if M1 doesn't add vitest tests.

3. **`buildFlashcardSlice` Strategy-3 fallback — apply mid-sprint or only for closed sprints?** Story §1.5 risk row says apply when "the planned date filter returns zero rows but git history shows in-window commits". My gotcha #6 above narrows it to closed sprints (`sprint_status === 'Completed'`) to avoid false positives during active execution. **Decision needed:** is the narrower closed-sprint-only gate acceptable, or should the Developer apply Strategy-3 unconditionally? **Recommendation: closed-only gate.** Mid-sprint, the Reporter-context bundle is rebuilt at sprint close anyway (close_sprint Step 3.5); applying Strategy-3 only at close-time is sufficient and avoids polluting the bundle with un-flashcarded commit subjects during active execution.

4. **Anomaly-detection threshold default (4× median)** — story §1.4 Open Q #2 marks this as `_accept recommended_` (4×). The blueprint encodes 4× as a `threshold = 4` parameter on `detectAnomalies`. **No decision needed; logging here so the Developer doesn't re-litigate.** If a future story wants to tune, the parameter is already there.
