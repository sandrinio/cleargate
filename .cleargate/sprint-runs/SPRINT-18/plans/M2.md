**Gotchas surfaced from code inspection:**

- **GOTCHA-1 (test-seam exit throws):** Sibling pattern at `sprint-archive.test.ts:25-29` shows `exitFn` throws `Error("exit:<n>")` to short-circuit the handler. The handler must `try/catch` around the throw at the test boundary OR the handler structure must call `exitFn` only at the top-level (after collecting all check results), as documented in flashcard `2026-04-20 #cli #test-seam #exit`. **Implementation rule:** run all 4 checks to completion BEFORE invoking `exitFn`. Do not let an early `exitFn` skip later checks — that would also break R2 (all 4 checks always run). The `await Promise.all([...])` shape in story §3.2 already enforces this; preserve it.
- **GOTCHA-2 (no top-level await in `sprint.ts`):** Per the file header at `sprint.ts:16` (FLASHCARD #tsup #cjs #esm), the source file forbids top-level await. The new `sprintPreflightHandler` MUST be a regular `async function` returning `Promise<void>` (or sync returning `void` if all checks can be sync — see GOTCHA-3). Do NOT add a top-level `await` for any module-init work.
- **GOTCHA-3 (sync vs async):** The four checks are all filesystem + child-process synchronous operations (`fs.existsSync`, `fs.readFileSync`, `execSync`/`spawnSync`). Story §3.2 sketches `Promise.all` + `await`, but synchronous `spawnSync` is the path of least surprise here and matches `sprintArchiveHandler` (line 761). Recommend `function sprintPreflightHandler(opts, cli?): void` returning `void` with `exitFn` at the bottom — mirrors `sprintCloseHandler:255` and `sprintInitHandler:91`. Story §3.2's async sketch is illustrative not normative; collapse to sync.
- **GOTCHA-4 (Commander v12 routing):** Per flashcard `2026-04-25 #cli #commander #subcommand-routing`, Commander v12 does NOT treat `<verb>` as a catch-all. The new `sprint preflight <sprint-id>` block MUST be a literal `.command('preflight <sprint-id>')`, NOT a parameterized fallthrough. Wire it identically to `sprint archive`/`sprint reconcile-lifecycle` (`cli.ts:351-371`).
- **GOTCHA-5 (vitest worker hygiene):** Per flashcards `2026-05-01 #vitest #leak #posttest` and `2026-05-01 #worktree #vitest #cleanup`, vitest tinypool workers leak ~3-4GB each across runs. Two implications: (a) `afterEach` MUST kill the test worktree's git worktree BEFORE removing its directory — order: `git worktree remove --force <path>` then `fs.rmSync(fixtureDir, {recursive: true, force: true})`. CR-023 ships `pool: 'forks'` durable fix during Wave 1; this story still lands in Wave 1 in parallel with CR-023, so the test file MUST self-clean even if pool config is mid-flight. (b) Do NOT spawn `npx vitest` in tests; the file is loaded by the existing vitest run. No issue here unless someone adds a sub-process invocation.
- **GOTCHA-6 (`git status --porcelain` semantics):** Story §1.2 R2 check 4 says "Run `git status --porcelain` against the `main` branch (current HEAD if HEAD is `main`; otherwise stash-aware check via `git diff main`)." The simpler implementation: assert HEAD points at `main` (`git symbolic-ref --short HEAD`); if not, fail check 4 with `not on main: HEAD is <branch>`; if yes, run `git status --porcelain` and assert empty. Do NOT implement a `git diff main` fallback unless QA finds a clear use case — feature creep beyond R2's literal text.
- **GOTCHA-7 (`git worktree list --porcelain` glob):** Per story §1.5 risk row, parse line-by-line for `worktree <path>` lines; match the path's basename against `STORY-*` AND ensure the parent dir basename is `.worktrees`. Strict glob `**/.worktrees/STORY-*` per story Open Question 2 (recommended accepted). Do not flag worktrees outside `.worktrees/`.
- **GOTCHA-8 (sprint-id regex):** Per story §3.3 API contract, `<id>` must match `^SPRINT-\d{2,3}$`. Implement `isValidSprintId()` as a private helper. Reject anything else with exit 2 (usage error, NOT exit 1 check failure). Story §1.4 Open Question 1 (typo'd id with no plan file) is also exit 2 per recommended-accepted; pin scope: validate the SHAPE only, NOT the existence of a plan file (existence-check is recommended-accepted but adds I/O — implement only the shape check, document the existence-check skip in a code comment, surface to orchestrator below).
- **GOTCHA-9 (branch-ref derivation):** `SPRINT-18` → `refs/heads/sprint/S-18`. Reuse `deriveSprintBranchForArchive()` at `sprint.ts:436-440` — extract to a non-private helper or duplicate as `deriveSprintBranch(sprintId)` (cleaner). Recommend rename to `deriveSprintBranch` and update the one existing call site at `sprint.ts:686`. **Open decision flagged below.**
- **GOTCHA-10 (prev-sprint id arithmetic):** `SPRINT-19` → prev = `SPRINT-18`. `SPRINT-01` → no prev (skip per scenario 8). Implementation: parse trailing digits with `/^SPRINT-(\d+)$/`, decrement, re-pad to 2 digits (`String(n-1).padStart(2, '0')`). For `SPRINT-100` → `SPRINT-099`? Story uses 2-3 digits per §3.3 regex; pad to 2 for the common case. Call out: SPRINT-100+ is theoretical — current pad-to-2 handles SPRINT-01..SPRINT-99 cleanly and gracefully degrades for 3-digit ids.

The recommended answer adds plan-file-existence I/O at usage-validation time. **Architect recommendation: implement shape-only first; add existence-check ONLY if scenario 7 ("Usage error on missing arg") expands to require it.** The Gherkin scenarios in §2.1 do NOT include a "typo'd sprint id with no plan file" scenario — only the missing-arg case (scenario 7). Implementing both is feature creep against the test surface. Developer should land shape-only (regex match) and put a `// TODO: existence-check per story §1.4 Q1` comment. Surface to orchestrator if QA flags.
