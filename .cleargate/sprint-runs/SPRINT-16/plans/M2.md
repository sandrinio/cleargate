## Wire-format reference (orientation only — single source of truth is EPIC-023 §2)

**Request (POST `cleargate_sync_work_items`):** `{ items: SyncItemPayload[] }` (max 100). Each `SyncItemPayload`: `cleargate_id, type, status, frontmatter, body, file_sha, last_synced_body_sha`. **Drop `project_id`** — JWT-derived server-side; do not duplicate in the body.

**Response:** `{ accepted: AcceptedItem[], conflicts: ConflictItem[], errors: ErrorItem[] }`. Field shapes per EPIC-023 §2.4 — do not redefine.

**Canonical sha:** `sha256(body + serializeFrontmatter(frontmatter))` where `serializeFrontmatter` is the function at `cleargate-cli/src/lib/frontmatter-yaml.ts:17` (CORE_SCHEMA, lineWidth -1, noRefs, noCompatMode, quotingType `"`, forceQuotes false, with `---\n…\n---` framing). Server replicates byte-identical at `mcp/src/utils/serialize-frontmatter.ts` (new helper introduced by STORY-023-02).

## Cross-story risks

1. **`commands/sync.ts` filename collision (HIGH).** STORY-023-01 specifies `cleargate-cli/src/commands/sync.ts` as the new file. That file exists (STORY-010-04, 718 lines, full pull/merge/push driver) and `cleargate sync` is already a registered Commander command (`cli.ts:550-562`). Authoring blind to this would cause one of: (a) wholesale replacement of the existing command (regression), (b) merge-conflict bounce. **Mitigation:** new command at `cleargate-cli/src/commands/sync-work-items.ts` exporting `syncWorkItemsHandler`; Commander registration is `cleargate sync work-items` (subcommand) OR a `--scope` branch on the existing handler. Orchestrator must confirm one of the two before STORY-023-01 spawns. See Open Decision #1.

2. **`js-yaml` dep missing in `mcp/`.** STORY-023-02 §3.2 imports `js-yaml`; mcp/package.json does not list it. Adding the dep is a `package.json` edit during STORY-023-02 — surfaces in PR diff. Lane stays standard (was already standard); no rubric impact. Single-version pin `^4.1.0` matches cleargate-cli.

3. **`project_id` in wire body conflicts with JWT-derived `ctx.project_id`.** EPIC-023 §2.3 declares `project_id: string` in the request body, but server context derives `project_id` from JWT claims (verified: `mcp/src/mcp/transport.ts:48`). If the CLI sends a `project_id` and the server ignores it, that's silent dead weight. If the server validates it against the JWT, that's an extra trust check the spec doesn't justify. **Resolution:** drop `project_id` from the wire request. Update EPIC-023 §2.3 + both stories' Zod schemas. Architect locks this; orchestrator confirms before merge.

4. **CLI/server frontmatter serialization parity (R-1 from EPIC-023 §5).** STORY-023-02 §3.2 shows server using `yaml.dump(fm, { sortKeys: true })`; CLI uses CORE_SCHEMA + extensive options + `---` framing. Different bytes → every sync produces "everything is dirty". **Resolution:** server reimplements `serializeFrontmatter` byte-identical at `mcp/src/utils/serialize-frontmatter.ts`. Integration test at the boundary (STORY-023-02 unit test #1) computes the sha both sides on a known fixture and asserts equality.

5. **Frontmatter key `parent_epic_ref` vs `parent_ref` drift** (FLASHCARD `2026-04-30 #frontmatter #hierarchy`). The sync walker reads every frontmatter — a Draft using either variant must round-trip losslessly. `parseFrontmatter` is YAML-via-CORE_SCHEMA so it's lossless; just verify a fixture with each variant in the test suite.

6. **`writeAtomic` rename race on Windows.** Not applicable (project is macOS/Linux). But the `.tmp.<timestamp>` collision during a quick re-run is mitigated by `Date.now()` suffix per `push.ts:390`.

7. **Sync-log entry per-item vs per-batch.** Existing `sync.ts:217` writes per-operation entries. Story §3.2 doesn't specify granularity. Recommendation: one sync-log entry per `cleargate sync` invocation summarizing accepted/conflicts/errors counts (not per-item) to avoid log explosion on 100-item batches. Surface as Open Decision #2.

8. **STORY-023-04 lane downgrade risk.** Story claims `fast`. Audit:
   - Files: `lib/admin-url.ts` + `test/lib/admin-url.test.ts` = 2 files. PASS (≤2 — note: the story's edit of `commands/sync.ts` is owned by STORY-023-01 per architect decision; this story stays at 2 files).
   - LOC: ≤50 net (function ~30 LOC, test ~50 LOC — tests count toward cap; total ~80 LOC). **FAIL the strict ≤50 cap.** Per rubric (rules §1) tests count. Recommend demote to `standard`. Surface as Open Decision #3.
   - Forbidden surfaces: none. PASS.
   - New dep: none. PASS.
   - Single Scenario: 7 scenarios. **FAIL.** Per rubric (rule §4) "exactly one Scenario block". Demote to `standard`.
   - **Net: STORY-023-04 lane should be `standard`, not `fast`.** Architect flags this; orchestrator decides whether to update story frontmatter pre-spawn.
