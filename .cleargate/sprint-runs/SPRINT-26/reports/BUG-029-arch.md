### 6. token-ledger.sh tuple-match logic correctness

CONFIRMED. The jq scan at line 143:

```
[scan("(?:STORY|PROPOSAL|EPIC|CR|BUG|HOTFIX)[-=][A-Za-z0-9]+(?:-[A-Za-z0-9]+)?")]
```

uses non-capturing groups `(?:...)` so `scan` returns the **full match string** (correct). With capture groups, scan would return only the group contents and lose the `STORY-` prefix. The character class `[-=]` admits both `STORY-NNN-NN` and `STORY=NNN-NN` dispatch-prompt formats, with post-normalization `sed 's/=/-/g'` converting to canonical hyphen form. The optional alphanumeric suffix `(?:-[A-Za-z0-9]+)?` correctly matches both digit-keyed IDs (STORY-002-03) and letter-suffix IDs (STORY-A, STORY-B used in fixtures).

Tuple-match iteration at lines 159–166: for-loop over `.dispatch-*.json`, extracts each candidate's `work_item_id`, increments `MATCH_COUNT` on equality. Single-match → consume; multi-match → log warning + fall back to newest-file (defensive — should not occur in practice, but graceful). Zero-match → fall back to newest-file with warning. All three branches log to `HOOK_LOG`.

The marker-consumption rename at line 188 (`mv "${DISPATCH_FILE}" "${DISPATCH_PROCESSED}"`) is unchanged from CR-026 — atomic claim, race-safe across concurrent SubagentStop fires.
