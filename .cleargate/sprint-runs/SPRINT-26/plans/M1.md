## Open decisions for orchestrator

These I will NOT decide; they need either user input or an Architect re-dispatch with more context:

1. **BUG-027 — sentinel-aware fallback chain ordering.** The Sprint Plan §2.5 ADR section endorses "fallback should consult `.cleargate/sprint-runs/.active` sentinel + most-recent dispatch-marker before grep; grep is a last resort". My code inspection confirms the dispatch-marker block (lines 117-141) already runs first. The question is **what comes between the dispatch-marker block and the legacy transcript-grep**:
   - **Option A:** Read the most-recent ledger row from `${LEDGER}` itself (the file the hook is about to append to) and reuse its `work_item_id`. Cheap; works for orchestrator-coordination calls because the prior subagent dispatch already wrote a row with the correct work_item_id.

   - **Option C:** Constrain the transcript-grep to ONLY scan transcript content authored by the orchestrator (skip system-message context, skip file-read tool results). Requires changes to the jq pipeline at `:245-257`.
   - **Recommendation (defer to user):** Option A. Cheapest, most accurate, avoids log parsing. But Architect has not verified that ledger reads are race-safe under concurrent SubagentStop fires — needs orchestrator confirmation OR a Read of `cleargate-cli/test/hooks/token-ledger-delta.test.ts` to inspect the `.session-totals.json` atomic-rename pattern for precedent.

Cheap, correct for the stated bug (user couldn't tell which file would change), but less informative for files where post-merge surgery yields a third hash.
