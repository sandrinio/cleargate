**Verb-to-status reconciliation map (v1):**

| Commit verb pattern | Applies to types | Expected terminal status | File location |
|---|---|---|---|
| `feat(STORY-NNN-NN): ...` | STORY | `Done` (per `TERMINAL_STATES = ['Done', 'Escalated', 'Parking Lot']` in `.cleargate/scripts/constants.mjs:45`) | `archive/` |

| `fix(BUG-NNN): ...` | BUG | `Verified` | `archive/` |
| `merge: <TYPE>-NNN → ...` | any | (ignored — merge commits don't carry expectations; the merged-in commit is authoritative) | n/a |
| `chore(SPRINT-NN): ...`, `chore: ...`, `file(...)`, `plan(...)`, `docs: ...`, `refactor: ...` | n/a | (no expectation — bookkeeping verbs) | n/a |
| `feat(BUG-NNN): ...` (verb mismatch) | BUG | soft warning ("verb 'feat' unusual for BUG; expected 'fix'") — **does not block** | n/a |

**Multi-ID parsing (CR §4 Scenario 3):**

- The parser scans the commit subject + first non-empty body line for ALL ID patterns matching `(STORY-\d{3}-\d{2}|CR-\d{3}|BUG-\d{3}|EPIC-\d{3}|PROPOSAL-\d{3}|PROP-\d{3}|HOTFIX-\d{3}-\d{2})`.
- Each ID gets its own validation against the verb→status map.
- **Bundled-commit grouping (sprint plan §2.3):** the `commit_shas: string[]` field on `DriftItem` deduplicates by SHA. A single bundled commit `feat(STORY-001-01)+CR-001 ...` produces ONE entry per stale ID, each entry having the bundled commit's SHA exactly ONCE in its `commit_shas` array. Two stale IDs in the same commit = TWO `DriftItem` entries (each with `commit_shas: [<sha>]`), NOT one entry with two SHAs.
- This guarantees the punch list cardinality matches the number of stale artifacts, not the cartesian product of (artifacts × commits).

**Carry-over semantics:**

- `carry_over: true` in an artifact's frontmatter → `reconcileLifecycle` skips that artifact silently (it does not appear in `drift` or `clean`).
- `carry_over: false` (or missing) → normal reconciliation applies.
- The flag is set **explicitly by the human at sprint close** (per CR §1 final paragraph) — not auto-inferred. No heuristic. The Architect/Developer/QA do not toggle it; the orchestrator does, on user instruction.
- `close_sprint.mjs` does NOT auto-promote `carry_over: true` artifacts forward into the next sprint's plan. That's a separate human action (edit the next sprint plan's frontmatter to include the artifact ID).

**Gotchas surfaced from code inspection:**
- `close_sprint.mjs` currently has no `imports` for the CLI runtime — it operates on `state.json` directly via `fs`. Calling `reconcileLifecycle` from `lifecycle-reconcile.ts` (TypeScript-source under `cleargate-cli/src/lib/`) requires either (a) the compiled `cleargate-cli/dist/lib/lifecycle-reconcile.js` (ESM) imported via `await import()` from the script, or (b) a thin CLI wrapper subcommand the script shells out to. **Recommend (b)** — `cleargate sprint reconcile-lifecycle` — because the script lives under `.cleargate/scripts/` (repo-internal) while the lib lives under `cleargate-cli/dist/` (per-package), and import-path resolution from a `.mjs` script to a sibling package's `dist/` is non-portable across worktrees.
- The `sprintInitHandler` at `cleargate-cli/src/commands/sprint.ts:70` is **v1-inert** (lines 84–86): when execution mode is v1, it prints inert message and exits 0 — never reaches the new gating code. **Confirm:** CR-017's gating only fires under v2 (which is what SPRINT-15 + SPRINT-16 use). Document this explicitly in protocol §25 / §26 — under v1 both gates are dormant.
- `sprintInitHandler` shells out via `spawnFn('bash', [runScript, ...])` (line 92). The two new gating calls must happen **before** that spawn, in the TypeScript handler, NOT in `init_sprint.mjs`. Reason: if gating fails, we want to exit before `init_sprint.mjs` mutates `state.json`.
- The `--allow-drift` flag must be added to the Commander option list at `cleargate-cli/src/cli.ts:248` — currently `sprint init` only declares `--stories`. Add `.option('--allow-drift', 'permit lifecycle drift at sprint kickoff (v1 warn-only); does NOT waive decomposition gate')`.
- Multi-ID parsing must use **non-greedy** matching to avoid eating `STORY-013-08` as `STORY-013` followed by `-08`. The regex `(STORY-\d{3}-\d{2}|...)` with the longest alternation first (per Bash tool guidance) is safe.
- Verb mismatch detection should NOT block in v1 — a soft warning to stderr is the v1 contract. v2 may escalate. Document the v2-escalation path in protocol §25.
- **Carry-over flag MUST be read from the artifact's CURRENT frontmatter (in pending-sync), not the historical version at the commit SHA.** Reconciliation is "where is this artifact NOW" — the human sets `carry_over: true` after the fact, not as part of the original commit.
