## `assert_story_files.mjs` extraction-strategy decision

**Decision: PATH (a) — lightweight shell-out + JSON-stdout protocol.** NOT path (b).

**Rationale (decision recorded for the Developer to follow verbatim):**
- Path (a) total cost: ~30 LOC. Add `export` keyword to two functions (lines 74, 90) + new `--emit-json` CLI flag block in `main()` that prints `{ workItemIds: [...] }` to stdout and exits 0; in `sprint.ts`, `execSync('node .cleargate/scripts/assert_story_files.mjs <sprint-file> --emit-json', { encoding: 'utf8' })` and `JSON.parse(stdout)`.
- Path (b) total cost: ~80 LOC. New `cleargate-cli/src/lib/work-item-extractor.ts`, refactor mjs to import via `createRequire` shim, update mirror under `cleargate-planning/`, new MANIFEST entry, regenerate scaffold prebuild. **Rejected because:** (i) the mjs script is invoked via `run_script.sh` from sprint init/close pipelines that pre-date the CLI bundle and have no TS toolchain in their hot path; (ii) `createRequire` of a TS-emitted module from raw mjs requires a published-CJS-or-bundled-ESM artifact, which forces a build-order coupling between the CLI prebuild and the bash script's runtime — exactly the kind of cross-package coupling protocol §13 cautions against; (iii) the helpers are 30 LOC of pure-regex extraction with zero external deps — duplication risk is near-zero, and the mjs already lives in BOTH live + canonical (verified `cleargate-planning/.cleargate/scripts/assert_story_files.mjs` exists).
- **Contract guard against drift** (the §3 risk row CR-027's spec calls out): the new vitest scenario #3 (per-item gate failure naming) asserts the `extractWorkItemIds` regex shape end-to-end through the shell-out path. If a future CR mutates the mjs regex without touching `sprint.ts`, scenario #3 fails fast.
- **Filename for the shell-out wrapper inside `sprint.ts`:** add a private helper `extractInScopeWorkItemIds(sprintFilePath, cwd, execFn)` adjacent to `checkPerItemReadinessGates` so the `execFn` test seam composes cleanly (same seam already used by `checkNoLeftoverWorktrees` at line 940-950). Tests inject a stub `execFn` that returns canned JSON; no real shell-out in unit tests.

**`assert_story_files.mjs` net edits (path-a):**
1. Line 74 prepend `export` → `export function extractWorkItemIds(text)`. (verified `function` keyword today).
2. Line 90 prepend `export` → `export function findWorkItemFile(repoRoot, workItemId)`. (verified `function` keyword today).

Add `--emit-json` flag handling in `main()` (after line 206 `args` parse): when `args.includes('--emit-json')`, parse the deliverables section, run `extractWorkItemIds`, write `JSON.stringify({ workItemIds })` to stdout, `process.exit(0)`.
