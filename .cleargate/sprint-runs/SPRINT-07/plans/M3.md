## STORY-010-04 section

### Canonical 6-step sync driver sequence (enforces R2)

`cleargate-cli/src/commands/sync.ts :: syncHandler(opts)`:

1. **Identity + sprint resolve** — `resolveIdentity(projectRoot)` (`identity.ts:101`) + `resolveActiveSprintDir(projectRoot)` (`sync-log.ts:64`).
2. **List remote updates** — `callMcp('cleargate_list_remote_updates', { since: wikiMeta.last_remote_sync ?? '1970-01-01T00:00:00.000Z' })` → `RemoteUpdateRef[]`. For each ref: `callMcp('cleargate_pull_item', { remote_id })` → `RemoteItem`. Accumulate in `pulled: RemoteItem[]`.
3. **(010-05 insertion point)** `// after pull, before detect`.
4. **Classify** — for each tracked local work-item with `remote_id` set, pair with `pulled[ref]` (or a fresh `cleargate_pull_item` for the push-side items). Build a `SinceLastSync` struct per §"SinceLastSync assembly" below. Call `classify(local, remote, since)` (`conflict-detector.ts:69`). Partition into maps by `resolution`.
5. **Resolve** — iterate in a **stable order**:
   - `resolution:'push'` / `'pull'` / `'no-op'` → queue for step 6 apply phase.
   - `'merge'` → `promptThreeWayMerge({ local, remote, base, itemId })` (`merge-helper.ts:118`). Apply `result.body` on `keep|take|edited`; on `aborted` → push this item into `conflictsJson` and continue.
   - `'merge-silent'` → apply both sides without prompt; sync-log `op:'sync-status'`, `result:'ok'`.
   - `'remote-wins'` → write remote status to local frontmatter; sync-log `op:'conflict-remote-wins'`.
   - `'refuse'` / `'halt'` → append to `conflictsJson`; sync-log `op:'conflict-refused'`, `result:'halted'`. DO NOT abort the loop — continue processing other items (AC §2.1 "remaining pulls/pushes after the conflict item still execute").
6. **Apply + log** — for each queued pull: overwrite body + frontmatter. For each queued push: `callMcp('push_item', …)`. Append one `sync-log` summary line per item. Update wiki-meta `last_remote_sync`. Atomic-write `.cleargate/.conflicts.json` (tmp + rename) with the accumulated unresolved items.

**R2 invariant (non-negotiable):** step 6 push calls MUST NOT begin until step 2's pulls are all returned/awaited. Enforce by structuring the driver as `const pulled = await pullAll(); … await applyPushesAndPulls();` in two distinct `await`ed phases. The test below asserts call order.

### MCP client reuse — there is none; build it once

**Contradiction flagged to orchestrator:** the SPRINT-07 sprint file + EPIC-010 §0 reference "existing push-item client"; the story text says "reuse existing MCP client plumbing". Grep confirms **no such client exists**: `cleargate-cli/src/` has `admin-api/client.ts` (admin REST only), `commands/join.ts` (bare `fetch` for `/join/:token`), and nothing that speaks MCP JSON-RPC over `/mcp`. MCP is exposed via `@modelcontextprotocol/sdk` `StreamableHTTPServerTransport` in stateless mode (`mcp/src/mcp/transport.ts:31-73`).

**Resolution:** 010-04 creates `cleargate-cli/src/lib/mcp-client.ts` once. Shape:

```ts
export interface McpClient {
  call<T>(tool: string, args: Record<string, unknown>): Promise<T>;
  adapterName(): Promise<'linear'|'jira'|'github-projects'|'no-adapter-configured'>;
}
export function createMcpClient(opts: {
  baseUrl: string;
  token: string;                  // JWT from refreshToken → access-token exchange
  fetch?: typeof globalThis.fetch;
}): McpClient;
```

Implementation uses `POST ${baseUrl}/mcp` with `Authorization: Bearer <jwt>` and a minimal JSON-RPC envelope (`{jsonrpc:'2.0', method:'tools/call', params:{name, arguments}, id}`), parsing the streamable-http response. Token acquisition: reuse `createTokenStore()` + call MCP's `/auth/refresh` (EPIC-003) to exchange refresh→access JWT; cache access JWT in memory for process lifetime. **Scope note:** a full access-token exchange flow lives in `STORY-004-08` (not yet shipped). For M3, 010-04 may require `CLEARGATE_MCP_TOKEN` env as a short-circuit if refresh-exchange is incomplete — **flag to orchestrator**.

### `cleargate-cli/src/lib/slug.ts`

**`slugify(title: string, max: number = 40): string`**
Algorithm (deterministic, no locale dependency):
1. NFKD-normalize then strip combining marks (`.normalize('NFKD').replace(/\p{M}/gu, '')`) so `é→e`, `ü→u`.
2. Lowercase.
3. Replace runs of `[^a-z0-9]+` with `-`.
4. Trim leading/trailing `-`.
5. Truncate to `max` chars; re-trim trailing `-`.

Edge cases:
- Empty / whitespace-only → returns `'untitled'` (story gives no example; lock as deterministic fallback).
- All-dash after sanitization (e.g. title `"!!!"`) → returns `'untitled'`.
- Unicode NOT in Latin-extended range (CJK, emoji) → stripped by step 3 → step 6 fallback. Matches story's "allow users — billing/accounts — to export CSVs & also PDFs !!!" example: expected `allow-users-billing-accounts-to-expor` (39 chars, trailing `-t` cut, re-trim leaves no trailing dash).

**`nextProposalId(projectRoot: string): Promise<number>`**
Scans both `.cleargate/delivery/pending-sync/` and `.cleargate/delivery/archive/` for any `.md` file whose frontmatter has `proposal_id: "PROP-NNN"` or unquoted `proposal_id: PROP-NNN` (both forms exist today — `grep` of archive confirms 6 quoted, 2 unquoted). Returns `max(N) + 1`. Gap-tolerant: returns max+1, **not** first gap (story §2.1 "nextProposalId scans both directories" + "result is 32" given max=31).
