**Schema / new files:**
- `.cleargate/.sync-marker.json` (runtime-created, gitignored):
  ```json
  { "last_check": "2026-04-19T20:00:00Z" }
  ```
  Minimum shape. `last_check` only. First-run semantics: if the file is missing, the hook writes `{ last_check: <now> }` and SKIPS the MCP call (24h grace period — story §1.2 is silent on this; decision made here to avoid nagging on first session of the day after `cleargate init`).

**`cleargate sync --check` contract (hook-safe):**
- Reads `.cleargate/.sync-marker.json.last_check`; falls back to `'1970-01-01T00:00:00.000Z'` if absent.
- Calls `mcp.adapterInfo()`; if `!configured || name === 'no-adapter-configured'` → print `{"updates":0,"error":"adapter-not-configured"}` and exit 0. Do NOT exit 2 as `syncHandler` does (that would be hook-hostile — R7).
- Calls `mcp.call<RemoteUpdateRef[]>('cleargate_list_remote_updates', { since })`.
- Prints ONE JSON line: `{"updates":<len>,"since":"<iso>"}`.
- On ANY thrown error → prints `{"updates":0,"error":"<message-first-200-chars>"}` and exits 0.
- Updates `.cleargate/.sync-marker.json.last_check = <now>` via atomic tmp-write+rename (mirror `writeAtomic` in `sync.ts:508`).
- Does NOT call `appendSyncLog`. Does NOT touch wiki meta, `.conflicts.json`, pull queue, push queue, or intake.
- Env precedence identical to `syncHandler` (reads `CLEARGATE_MCP_TOKEN` + `CLEARGATE_MCP_URL`); if either is missing → treat as error path above (silent-safe output, exit 0).

**SessionStart hook nudge — behavioral contract:**
- Runs AFTER the existing `doctor --session-start` call (do not replace).
- `now=$(date +%s)`; read `last_check` from `${REPO_ROOT}/.cleargate/.sync-marker.json` (use `node -e` inline or `grep -oE` — Developer's call, simplest). If file missing → write `{ "last_check": "<now-iso>" }` and exit 0 silently. No MCP call.
- If `(now - last_check_epoch) < 86400` → exit 0 silent. No MCP call.
- If ≥24h AND an active sprint exists (decision: probe by checking that `resolveActiveSprintDir` would return a non-`_off-sprint` directory — invoke via `cleargate sync --check` and let it return `updates:0` if nothing is active; do NOT duplicate sprint-detection logic in bash).
- Invoke `timeout 3 node "${REPO_ROOT}/cleargate-cli/dist/cli.js" sync --check 2>/dev/null`. Timeout command required (R7).
- Parse `updates` field with `node -e` or a `grep -oE '"updates":[0-9]+'` + cut; on `updates > 0` print exactly:
  `ClearGate: N remote updates since yesterday — run \`cleargate sync\` to reconcile.`
  (Story §1.2 uses a 📡 emoji; this plan drops the emoji — flashcard `#no-emoji` is implicit from CLAUDE.md global instruction "MUST avoid using emojis". Open question #1 surfaced below.)
- ALWAYS exit 0. ALWAYS update the marker's `last_check` (done by `sync --check` itself — hook does not re-write it).
- On timeout / nonzero exit / malformed JSON → silent; exit 0.

**§14 outline (9 non-negotiable rules — Developer writes verbatim):**

Each rule: `### §14.N <Title>` heading, one-line imperative rule, 2–3 sentence rationale citing shipped code. Cap each rule at ~80 words. Total §14 ≤ 900 words.

1. **§14.1 Sync matrix & authority split.** *Rule:* Remote is authoritative for status/assignees/comments; local is authoritative for work-item body. *Source:* EPIC-010 §4 table + `cleargate-cli/src/commands/sync.ts:282-288` (conflict-detector snapshot shape).
2. **§14.2 Conflict resolution.** *Rule:* content+content → 3-way merge prompt; status+status → remote-wins silently; delete+edit (either direction) → refuse; unrecognized conflict shape → `halt` (not guess). *Source:* `cleargate-cli/src/lib/conflict-detector.ts` matrix (8 states + `unknown` halt); `cleargate-cli/src/commands/sync.ts:307-367`.
3. **§14.3 Sync ordering invariant.** *Rule:* `cleargate sync` MUST execute pull → classify → resolve → push in that order; reversal amplifies conflicts. *Source:* `cleargate-cli/src/commands/sync.ts:6-16` (6-step driver doc comment); R2 mitigation test at `cleargate-cli/test/commands/sync*.test.ts`.
4. **§14.4 Identity resolution precedence.** *Rule:* `.cleargate/.participant.json` → `CLEARGATE_USER` env → `git config user.email` → `host+user`. Per-repo, not global. *Source:* `cleargate-cli/src/lib/identity.ts`; R5 mitigation.
5. **§14.5 Stakeholder-authored proposal flow.** *Rule:* Remote items with label `cleargate:proposal` and no local counterpart land in `pending-sync/PROPOSAL-*-remote-*.md` with `source: "remote-authored"`, `approved: false`. Label configurable via `CLEARGATE_PROPOSAL_LABEL` env. *Source:* `cleargate-cli/src/lib/intake.ts`; `cleargate-cli/src/commands/sync.ts:167-183`.
6. **§14.6 Comment policy.** *Rule:* Comments pull as read-only snapshots, active items only (current sprint + last 30d), rendered under `## Remote comments` with literal-string delimiters. Never pushed upstream. *Source:* `cleargate-cli/src/lib/wiki-comments-render.ts`; `active-criteria.ts`; R4/R6 mitigations.
7. **§14.7 Push preconditions.** *Rule:* `cleargate_push_item` requires `payload.approved === true` unless caller passes `skipApprovedGate: true` (reserved for `sync_status` internal callers). `pushed_by` is stamped from `members.email` via JWT `sub` → member lookup, NOT the JWT `sub` value itself. *Source:* `mcp/src/tools/push-item.ts:39,91-94,111-120,172,209`; flashcard `#mcp #jwt #attribution`.
8. **§14.8 Revert policy.** *Rule:* `cleargate push --revert <id>` soft-reverts via `cleargate_sync_status` status push (`archived-without-shipping`); never deletes the remote item. Refuses when local `status: done` unless `--force` is passed. *Source:* `cleargate-cli/src/cli.ts:268-273`; `cleargate-cli/src/commands/push.ts` revert branch.
9. **§14.9 Sync cadence.** *Rule:* All sync actions are manual (`cleargate sync` / `pull` / `push`). SessionStart hook SUGGESTS via `cleargate sync --check` — never auto-pulls, never auto-pushes. Throttle: ≤1 MCP probe per 24h per repo. *Source:* this story (`.claude/hooks/session-start.sh`); `.cleargate/.sync-marker.json`; R7 mitigation.
