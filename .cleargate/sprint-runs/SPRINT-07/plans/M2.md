## R1 — `PmAdapter` v1 interface: cross-PM-tool validation

**Methodology:** diffed the 4 verbs against Linear GraphQL (`@linear/sdk` `client.issue`, `client.issues`, `client.comments`), Jira REST v3 (`/rest/api/3/issue/{key}`, `/rest/api/3/search?jql=labels=...`), and GitHub Projects v2 GraphQL (`ProjectV2Item` with polymorphic `content: Issue|PullRequest|DraftIssue`).

**Top-level contract (must be PM-tool-agnostic):**
- All IDs are opaque `string`.
- All timestamps are ISO-8601 UTC `string` (normalise at adapter boundary; never leak `Date` objects).
- All optional fields are `T | null` — never `undefined` in the wire shape, per M1 lock on `last_remote_update: string | null`.
- Anything that is not 1:1 expressible across all three tools goes into `RemoteItem.raw: unknown`.

### Per-verb findings

**`pullItem(remoteId: string): Promise<RemoteItem | null>`** — returns `null` on not-found (do NOT throw for 404; reserve throws for transport/auth failures). All three tools return a single-resource shape. **Safe to abstract.** Linear returns `Issue.{id,identifier,title,description,state.name,assignees,labels,updatedAt}`; Jira returns `{key,fields.summary,fields.description,fields.status.name,fields.assignee,fields.labels,fields.updated}`; GH returns `ProjectV2Item.content.{title,body,state,assignees,labels,updatedAt}`. Map all to `RemoteItem`; stash native object under `.raw`.

**`listUpdates(since: string): Promise<RemoteUpdateRef[]>`** — lightweight poll. **Safe to abstract.** Linear: `client.issues({ filter: { updatedAt: { gte: since } } })` → `{id, updatedAt}[]`. Jira: `/search?jql=updated >= "<iso>"` with `fields=updated` → `{key, fields.updated}[]`. GH: `ProjectV2.items` with cursor; filter client-side by `updatedAt >= since` (no server-side filter on Project items — acceptable, API returns ≤100/page). Return shape `{ remote_id, updated_at }` is the common minimum.

**`pullComments(remoteId: string): Promise<RemoteComment[]>`** — read-only. **Safe to abstract with one caveat:** GH Projects v2 comments live on the **underlying Issue**, not on the `ProjectV2Item`. Adapter for GH must resolve `ProjectV2Item → content.id → issue.comments`. This is a GH-adapter concern; the interface stays `(remoteId: string) => RemoteComment[]`. Linear + Jira: direct 1:1. `author_email` is **not guaranteed** on GH (Users expose `login` + optionally `email` depending on privacy); return `null` when missing. Lock `author_email: string | null`.

**`detectNewItems(filter: { label?: string; since?: string }): Promise<RemoteItem[]>`** — the one verb where PM-tool shapes genuinely diverge, but the divergence is absorbable. Linear labels are first-class on `Issue`; Jira labels are string array on `fields.labels`; GH Projects v2 labels are on the **underlying Issue, not the Project item**. For GH, `detectNewItems({ label })` means "Project items whose underlying Issue has label L" — this is an extra hop but expressible. **Safe to abstract.** Server-side dedup (per B3) against `items.remote_id` column belongs in the endpoint handler, NOT the adapter — adapter returns everything matching the filter; handler excludes local-known IDs.

### Final TypeScript shapes (land verbatim in `mcp/src/adapters/pm-adapter.ts`)

```ts
export type PmToolName = 'linear' | 'jira' | 'github-projects';

export interface RemoteItem {
  remote_id: string;            // opaque; Linear "identifier" (LIN-1042), Jira "key", GH node ID
  title: string;
  body: string | null;          // description / body; may be empty
  status: string;               // remote-native workflow state name (not normalised)
  assignees: string[];          // emails when available; login/name when not
  labels: string[];
  updated_at: string;           // ISO-8601 UTC
  source_tool: PmToolName;
  raw: unknown;                 // native API object; consumers MUST NOT rely on shape
}

export interface RemoteUpdateRef {
  remote_id: string;
  updated_at: string;           // ISO-8601 UTC
}

export interface RemoteComment {
  id: string;                   // opaque remote comment ID
  author_email: string | null;  // null when tool doesn't expose email (GH privacy)
  author_name: string;          // always populated (display name or login)
  body: string;
  created_at: string;           // ISO-8601 UTC
  remote_id: string;            // parent item's remote_id
}

export interface PmAdapter {
  readonly name: PmToolName;
  pullItem(remoteId: string): Promise<RemoteItem | null>;
  listUpdates(since: string): Promise<RemoteUpdateRef[]>;
  pullComments(remoteId: string): Promise<RemoteComment[]>;
  detectNewItems(filter: { label?: string; since?: string }): Promise<RemoteItem[]>;
}
```

Differences vs. story §1.2 draft:
1. `pullItem` returns `RemoteItem | null` (not `RemoteItem`). Rationale: 404 is a non-exceptional control flow for `detectNewItems` + `cleargate pull <ID>`; throwing forces every caller into try/catch.
2. `RemoteItem.body: string | null` (not `string`). Jira issues routinely have empty description; GH draft issues ditto.
3. `RemoteComment.author_email: string | null` + new `author_name: string`. GH Projects privacy — see above.
4. `RemoteItem.source_tool` is locked to the `PmToolName` enum, not free-string.
5. `name` is `readonly`.

These adjustments keep the Linear concrete clean while not baking Linear assumptions into the generic. Dev A: land these shapes; do NOT re-litigate them in the PR.

### `conflict-detector.ts` exact shapes

```ts
export type ConflictState =
  | 'no-change'
  | 'local-only'            // local content edit, no remote change
  | 'remote-only'           // remote status/metadata change, local untouched
  | 'content-content'       // both bodies diverged since last sync
  | 'content-status'        // local body + remote status
  | 'status-status'         // both statuses diverged
  | 'local-delete-remote-edit'
  | 'remote-delete-local-edit'
  | 'unknown';              // R3: explicit fallthrough — never silently wrong

export type Resolution =
  | 'push'
  | 'pull'
  | 'merge'                 // three-way merge prompt
  | 'merge-silent'          // content+status: no prompt, apply both sides
  | 'remote-wins'           // status+status: silent remote takes, log conflict-remote-wins
  | 'refuse'                // halt, surface to human
  | 'halt';                 // unknown state — halt sync with explicit message

export interface LocalSnapshot {
  updated_at: string;
  body_sha: string;
  status: string;
  deleted: boolean;
}
export interface RemoteSnapshot {
  updated_at: string;
  body_sha: string;
  status: string;
  deleted: boolean;
}
export interface SinceLastSync {
  last_pushed_at: string | null;
  last_pulled_at: string | null;
  last_remote_update: string | null;  // lock per M1: string | null, opaque ISO
  last_body_sha: string | null;        // the sha at last successful sync (for merge-base)
}

export interface Classification {
  state: ConflictState;
  resolution: Resolution;
  reason: string;  // human-readable, used for sync-log detail + halt messages
}

export function classify(
  local: LocalSnapshot,
  remote: RemoteSnapshot,
  since: SinceLastSync,
): Classification;
```

### 8-state + 9th fallthrough decision table (PROP-007 §2.3 verbatim + R3 extension)

| # | State | Detection rule (exact) | Resolution | Reason string |
|---|---|---|---|---|
| 1 | `no-change` | `local.body_sha === since.last_body_sha && remote.body_sha === since.last_body_sha && local.status === remote.status && !local.deleted && !remote.deleted` | `pull` (no-op) | "no change since last sync" |
| 2 | `local-only` | `local.body_sha !== since.last_body_sha && remote.body_sha === since.last_body_sha && remote.status === local.status && !remote.deleted` | `push` | "local content edit only" |
| 3 | `remote-only` | `local.body_sha === since.last_body_sha && (remote.status !== local.status OR remote.updated_at > (since.last_pulled_at ?? '0'))` and bodies match | `pull` | "remote status/metadata change only" |
| 4 | `content-content` | `local.body_sha !== since.last_body_sha && remote.body_sha !== since.last_body_sha && !local.deleted && !remote.deleted` | `merge` | "both bodies diverged — three-way merge required" |
| 5 | `content-status` | `local.body_sha !== since.last_body_sha && remote.body_sha === since.last_body_sha && remote.status !== local.status` | `merge-silent` | "local body edit + remote status change — merged without prompt" |
| 6 | `status-status` | `local.body_sha === since.last_body_sha && remote.body_sha === since.last_body_sha && local.status !== remote.status && local.status !== <status at last sync>` (needs `since.last_status` — see note below) | `remote-wins` | "status diverged on both sides; remote authoritative" |
| 7 | `local-delete-remote-edit` | `local.deleted && remote.updated_at > (since.last_pulled_at ?? '0')` | `refuse` | "local deletion conflicts with remote edit" |
| 8 | `remote-delete-local-edit` | `remote.deleted && local.updated_at > (since.last_pushed_at ?? '0')` | `refuse` | "remote deletion conflicts with local edit" |
| 9 | `unknown` (R3) | any input combination not matched by rules 1–8 | `halt` | "conflict shape not recognized — please resolve manually and file a ClearGate bug with this sync-log entry" |

**Schema gap in `SinceLastSync`:** rule 6 needs `last_status` (the status recorded at last successful sync) to distinguish `status-status` from `remote-only`. M1 did not reserve this frontmatter field. Dev B: **add `last_synced_status: string | null` to `SinceLastSync` and document** that STORY-010-04 is responsible for populating it from frontmatter on each sync. If the orchestrator wants the field stamped into templates, that's a template-schema change and must go back to M1 — flag below.

   Call `openInEditor(tmpPath, { editor: opts.editor })`. On close, read back file, run `containsConflictMarkers(content)` — if true, reprompt (user didn't resolve). If false, `body = content; resolution = 'edited'`. Always `fs.unlink` the temp file before returning (even on error).
5. Never writes to the caller's file. Caller applies `MergeResult.body` to the frontmatter/body it owns.
