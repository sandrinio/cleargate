### D1 — Invites table schema (final Drizzle shape)

Add to `/Users/ssuladze/Documents/Dev/ClearGate/mcp/src/db/schema.ts` (append; do not reorder existing tables):

```ts
export const invites = pgTable(
  'invites',
  {
    id: uuid().primaryKey().defaultRandom(),           // the opaque invite token
    projectId: uuid()
      .notNull()
      .references(() => projects.id, { onDelete: 'cascade' }),
    memberId: uuid()
      .notNull()
      .references(() => members.id, { onDelete: 'cascade' }),
    email: text().notNull(),                           // denormalized from members.email, audit-stable
    role: text().notNull(),                            // 'user' | 'service' — for future role-at-time-of-invite audit
    createdBy: uuid()
      .notNull()
      .references(() => adminUsers.id),                // admin who issued the invite
    createdAt: timestamp({ withTimezone: true }).defaultNow().notNull(),
    expiresAt: timestamp({ withTimezone: true }).notNull(),
    consumedAt: timestamp({ withTimezone: true }),     // NULL = pending; set exactly once at redemption
  },
  (t) => [
    // Admin UI "list pending invites for project P" (EPIC-006) — partial index avoids consumed/expired rows.
    index('idx_invites_project_active').on(t.projectId).where(sql`${t.consumedAt} IS NULL`),
    // Redemption path hits the PK directly; explicit index not needed.
  ],
);
export type Invite = typeof invites.$inferSelect;
```

Rationale:
- `id: uuid().defaultRandom()` — doubles as the invite token (UUIDv4). Server never mints a separate secret; the row id IS the token. Matches STORY-004-07 §1.1 verbatim. See D2 for token-format rationale.
- `email` and `role` are denormalized on purpose. If a member is renamed/deleted pre-redemption, the invite still has the exact shape the admin issued (future audit/UI needs). Cascade FK cleans up on member deletion.
- `createdBy` traces the issuing admin (absent in SPRINT-02's Redis shape; cheap to add now, painful to backfill later).
- Partial index `WHERE consumed_at IS NULL` keeps admin UI queries O(log active) instead of O(log total). Uses `sql` template — Drizzle supports partial indexes via `.where()`.
- No UNIQUE on `(project_id, member_id)` — members-table unique `(project_id, email)` already prevents dup invites at insert time (handler rejects 409 before reaching `invites`). Allowing multiple invite rows per member is actually useful if we ever add "resend invite" (a new row with a fresh `id` + `expires_at`; old row is orphaned but not consumed). Out of scope for M4.
- **No extra `status` column.** `(consumed_at, expires_at)` derives all three UI states (pending / consumed / expired). Derived in SQL in the list query — see D4.

### D2 — Invite-token format: UUIDv4 (PK = token)

Locked by story file: STORY-004-07 §1.1 L52 — "Opaque UUIDv4 (random, not v7), matches EPIC-005 §6.4 resolution." Decision rationale retained for reference:

- **Not 32-byte base64url.** W3's admin API tokens (`mcp/src/admin-api/tokens.ts:11-12`) use `base64url(randomBytes(32))` for bcrypt-hashed long-lived API tokens. Different domain: API tokens must be wide enough to survive bcrypt's 72-byte clamp and live for months. Invites live 24h, are redeemed once, and are stored in plaintext as a PK. A 122-bit UUIDv4 against a 10/15min anonymous rate limit (D6) is brute-force-infeasible (10^7 years at 10 guesses/15 min).
- **Not v7.** v7 leaks creation timestamp in the prefix. Invite tokens are pasted into chat/email — time-leak is low but nonzero. Pick v4.
- **Not JWT.** Self-description is pointless for a PK that must be looked up in DB anyway; JWT cost is pure overhead.

`id: uuid().defaultRandom()` in Drizzle emits `gen_random_uuid()` in Postgres (pgcrypto), which is v4. Matches STORY-004-07 §1.1 verbatim.

### D3 — Redemption atomicity + error shape

#### D3-A — Redemption SQL

Single query from STORY-003-13 §1:
```sql
UPDATE invites
   SET consumed_at = now()
 WHERE id = $1
   AND consumed_at IS NULL
   AND expires_at > now()
 RETURNING member_id, project_id, role
```
If `rowCount === 1`, redemption succeeded. If `rowCount === 0`, run a second cheap query to distinguish cases:
```sql
SELECT consumed_at, expires_at FROM invites WHERE id = $1
```
- no row → **404** `{ error: 'not_found' }`
- row with `consumed_at IS NOT NULL` → **410** `{ error: 'invite_already_consumed' }`
- row with `expires_at <= now()` and `consumed_at IS NULL` → **410** `{ error: 'invite_expired' }`
- anything else (would be a bug) → 500

Both error conditions return **410 Gone** with distinct `error` strings. Story file L58-65 mandates 410 for both "expired" and "already consumed" (single status, distinguishable body). 404 is reserved strictly for "token not in table at all" (brute-force probe; attacker doesn't learn whether the token ever existed vs was consumed).

The status-code choice (410 vs 409) is locked to **410 for both** per story Gherkin L58-65. Rationale: 410 = "gone forever" correctly describes both (the invite is non-redeemable; consumed invites can't be resurrected). 409 (Conflict) would imply a retry could work.

### D4 — GET /members list rewrite: three statuses, SQL-derived

Current `members.ts:32-42` emits `status: 'pending' | 'active'` only (Redis key present → pending). After 004-07 the status must derive from `(consumed_at, expires_at)` across the JOINed `invites` row. Status values per STORY-004-07 §1.2 + §2 Gherkin L90-93: **`pending` | `active` | `expired`**.

Drizzle query shape (replaces members.ts:85-91):
```ts
const rows = await deps.db.execute(sql`
  SELECT m.id, m.project_id, m.email, m.role, m.display_name, m.created_at,
         CASE
           WHEN i.id IS NULL                                    THEN 'active'
           WHEN i.consumed_at IS NOT NULL                       THEN 'active'
           WHEN i.expires_at <= now()                           THEN 'expired'
           ELSE 'pending'
         END AS status
    FROM members m
    LEFT JOIN invites i ON i.member_id = m.id
                        AND i.consumed_at IS NULL
                        AND i.expires_at > now()
   WHERE m.project_id = ${pid}
`);
```
The LEFT JOIN filters on `consumed_at IS NULL AND expires_at > now()` so the join only matches **pending** invites. The CASE then distinguishes `pending` (join matched) from `expired` (join didn't match but a consumed-null expired row exists). **Gap in that logic:** if the join doesn't match because the only row is expired, the LEFT JOIN yields NULL → CASE routes to 'active'. To catch 'expired', we need an additional predicate: a second LEFT JOIN or a subquery. Practical fix:

```ts
SELECT m.id, m.project_id, m.email, m.role, m.display_name, m.created_at,
       CASE
         WHEN pending.id IS NOT NULL                            THEN 'pending'
         WHEN any_invite.id IS NOT NULL
              AND any_invite.consumed_at IS NULL
              AND any_invite.expires_at <= now()                THEN 'expired'
         ELSE 'active'
       END AS status
  FROM members m
  LEFT JOIN invites pending ON pending.member_id = m.id
                            AND pending.consumed_at IS NULL
                            AND pending.expires_at > now()
  LEFT JOIN LATERAL (
    SELECT id, consumed_at, expires_at
      FROM invites
     WHERE member_id = m.id
     ORDER BY created_at DESC
     LIMIT 1
  ) any_invite ON true
 WHERE m.project_id = ${pid}
```

`any_invite` is the most recent invite per member (LATERAL subquery). Status rules:
- Most recent invite is pending → `pending`.
- Most recent invite is expired (consumed_at IS NULL AND expires_at <= now()) → `expired`.
- Most recent invite is consumed → `active`.
- No invite rows exist for the member → `active` (edge case: member was seeded directly, e.g. the admin-self-insert in `mcp/src/admin-api/projects.ts:71-89`).

Developer implements this as a single `deps.db.execute(sql)` raw query (Drizzle's LATERAL support is limited) or via two queries + JS join. Architect picks **raw SQL** — performance is fine (single query, indexed on `member_id`), and the alternative (N+1 per member) is worse.

### D6 — Rate-limit bucket for `/join/:invite_token`

Current `mcp/src/middleware/rate-limit.ts:27-43` only operates on authenticated requests (`if (!request.claims) return`). No anonymous bucket exists. Sprint file L112 explicitly says: "If the anonymous rate-limit bucket doesn't exist, add it in STORY-003-13 rather than splitting a new story (~10 LOC)."

**Locked: STORY-003-13 adds a second middleware `buildAnonymousRateLimit(redis, {limit, windowSec})` in the same file.** Signature:

```ts
export interface AnonymousRateLimitOptions {
  limit: number;       // requests per window
  windowSec: number;   // window size
}

export function buildAnonymousRateLimit(
  redis: Redis,
  opts: AnonymousRateLimitOptions,
): preHandlerHookHandler {
  return async (request, reply) => {
    // Key by IP. Fastify sets request.ip from X-Forwarded-For when trustProxy=true
    // (already enabled in server.ts:50). Fall back to socket address.
    const ip = request.ip || 'unknown';
    const windowId = Math.floor(Date.now() / 1000 / opts.windowSec);
    const key = `rl:anon:${ip}:${windowId}`;
    const current = await redis.incr(key);
    if (current === 1) await redis.expire(key, opts.windowSec + 1);
    if (current > opts.limit) {
      return reply
        .code(429)
        .header('Retry-After', String(opts.windowSec))
        .send({ error: 'rate_limited', limit: opts.limit, window_seconds: opts.windowSec });
    }
  };
}
```

Starting values: **`limit: 10, windowSec: 900` (10 requests per 15 minutes per IP).** Rationale:
- Invite token is 122-bit UUIDv4. At 10 probes / 900s = 0.011 probes/sec, brute-force expected time is 2^121 / 0.011s ≈ 10^37 years. Room to increase later.
- Real users hit the endpoint **once** successfully. Legitimate retries (network, stale URL paste) exhaust in 2-3 calls. 10/15min is forgiving without being lax.
- Conservative start matches "production hardening" sprint theme; loosen in EPIC-006 if ops surfaces false positives.

Values are **hardcoded in STORY-003-13's route registration**, not plumbed through `config.ts`, to avoid introducing a new env var (sprint DoD L105 "no new env vars"). If ops needs to tune, M5 or later can promote to `RATE_LIMIT_ANON_JOIN_PER_15MIN`.

### D9 — CLI URL parsing (client-side)

`cleargate join <invite-url>` accepts both:
- **Full URL**: `https://mcp.example.com/join/<uuid>` — parses with `new URL()`; validates path matches `^/join/[0-9a-f-]{36}$`; extracts token from the last path segment; extracts base URL as `url.origin`.
- **Bare token**: `<uuid>` — validates UUIDv4 regex; requires `mcpUrl` from config (flags > env > config file, already shipped in W1). Errors cleanly if `mcpUrl` unset.

Other formats rejected with clear stderr. UUID regex: `/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i` (v4-compliant).

**Rationale for accepting bare token:** admin CLI prints the full URL (STORY-005-02 shipped); most users paste that. But documentation and tests should assert both forms; future deep-link UX (e.g. `cleargate://join/...`) can be added without breaking the bare-token path.

**Base URL extraction gotcha:** when user passes a full URL, `url.origin` is used as `mcpUrl` for the redemption POST — this overrides config. Same URL's base is then persisted into `~/.cleargate/config.json`? **No — do not persist automatically.** Persisting on `join` is a separate UX decision (W3 deliberately kept config writes manual). User sets `mcpUrl` via `--mcp-url` flag or env for subsequent commands. If they pass full URL on `join`, the redemption request uses it, but config stays as-is. Documented in stderr success message (see D10).

```ts
export async function registerJoinRoute(app: FastifyInstance, deps: JoinRouteDeps): Promise<void> {
  const anonLimit = buildAnonymousRateLimit(deps.redis, { limit: 10, windowSec: 900 });

  await app.register(async (scoped) => {
    scoped.addHook('preHandler', anonLimit);

    scoped.post<{ Params: { invite_token: string } }>(
      '/join/:invite_token',
      async (request, reply) => {
        const token = request.params.invite_token;

        // Atomic redemption
        const redeemed = await deps.db.execute(sql`
          UPDATE invites
             SET consumed_at = now()
           WHERE id = ${token}
             AND consumed_at IS NULL
             AND expires_at > now()
          RETURNING member_id, project_id, role
        `);

        if (redeemed.rows.length === 0) {
          // Disambiguate 404 vs 410
          const lookup = await deps.db.execute(sql`
            SELECT consumed_at, expires_at FROM invites WHERE id = ${token}
          `);
          if (lookup.rows.length === 0) {
            return reply.code(404).send({ error: 'not_found' });
          }
          const row = lookup.rows[0]!;
          if (row.consumed_at !== null) {
            return reply.code(410).send({ error: 'invite_already_consumed' });
          }
          return reply.code(410).send({ error: 'invite_expired' });
        }

        const { member_id: memberId, project_id: projectId, role } = redeemed.rows[0]!;

        // Fetch project name (single JOIN)
        const projRes = await deps.db.execute(sql`
          SELECT p.name AS project_name
            FROM projects p WHERE p.id = ${projectId}
        `);
        const projectName = projRes.rows[0]!.project_name;

        // Mint refresh token
        const refresh = await deps.jwt.issueRefresh({
          sub: memberId,
          project_id: projectId,
          role,
        });

        // Audit (best-effort)
        await writeAudit(deps.db, {
          projectId, memberId,
          tool: 'join', result: 'ok',
          ipAddress: request.ip ?? null,
          userAgent: request.headers['user-agent'] ?? null,
        });

        return reply.code(200).send({
          refresh_token: refresh.token,
          refresh_token_expires_in: deps.config.REFRESH_TOKEN_TTL_DAYS * 86_400,
          project_id: projectId,
          project_name: projectName,
          member_id: memberId,
          member_role: role,
        });
      },
    );
  });
}
```

```ts
import * as os from 'node:os';

const UUID_V4_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i;

export interface JoinOptions {
  inviteUrl: string;
  profile: string;
  mcpUrlFlag?: string;
  fetch?: typeof globalThis.fetch;                 // test seam
  createStore?: typeof createTokenStore;            // test seam
  hostname?: () => string;                          // test seam
  stdout?: (s: string) => void;
  stderr?: (s: string) => void;
  exit?: (code: number) => never;
}

export async function joinHandler(opts: JoinOptions): Promise<void> {
  const fetchFn = opts.fetch ?? globalThis.fetch;
  const stdout = opts.stdout ?? ((s) => process.stdout.write(s));
  const stderr = opts.stderr ?? ((s) => process.stderr.write(s));
  const exit = opts.exit ?? ((c) => process.exit(c));
  const hostname = opts.hostname ?? (() => os.hostname());

  // Parse input → { token, baseUrl }
  let token: string;
  let baseUrl: string;
  try {
    if (UUID_V4_RE.test(opts.inviteUrl)) {
      token = opts.inviteUrl;
      const cfg = loadConfig({ flags: { profile: opts.profile, mcpUrl: opts.mcpUrlFlag } });
      if (!cfg.mcpUrl) {
        stderr('cleargate: bare invite token requires mcpUrl. Pass --mcp-url <url> or set CLEARGATE_MCP_URL.\n');
        exit(5);
      }
      baseUrl = cfg.mcpUrl!;
    } else {
      const url = new URL(opts.inviteUrl);
      const m = url.pathname.match(/^\/join\/([0-9a-f-]{36})$/i);
      if (!m || !UUID_V4_RE.test(m[1]!)) throw new Error('bad path');
      token = m[1]!;
      baseUrl = url.origin;
    }
  } catch {
    stderr('cleargate: invalid invite URL or token format.\n');
    exit(5);
  }

  // POST redemption
  let response: Response;
  try {
    response = await fetchFn(`${baseUrl}/join/${token}`, { method: 'POST' });
  } catch (err) {
    stderr(`cleargate: cannot reach ${baseUrl} (${err instanceof Error ? err.message : String(err)}).\n`);
    exit(2);
  }

  if (response.status === 410) {
    const body = (await response.json().catch(() => ({}))) as { error?: string };
    if (body.error === 'invite_expired') {
      stderr('cleargate: invite expired. Request a new invite.\n');
    } else {
      stderr('cleargate: invite already consumed. Request a new invite.\n');
    }
    exit(3);
  }
  if (response.status === 404) {
    stderr('cleargate: invite not found.\n');
    exit(4);
  }
  if (response.status === 429) {
    const retry = response.headers.get('retry-after') ?? '900';
    stderr(`cleargate: too many requests. Retry after ${retry}s.\n`);
    exit(8);
  }
  if (response.status >= 500) {
    stderr(`cleargate: server error ${response.status}.\n`);
    exit(6);
  }
  if (!response.ok) {
    stderr(`cleargate: unexpected status ${response.status}.\n`);
    exit(7);
  }

  // 200 — extract refresh_token with named field access ONLY (D12)
  let body: unknown;
  try {
    body = await response.json();
  } catch {
    stderr('cleargate: server returned non-JSON response.\n');
    exit(7);
  }
  // Validate shape minimally (Zod)
  const b = body as {
    refresh_token?: string;
    project_name?: string;
    member_role?: string;
  };
  if (typeof b.refresh_token !== 'string' || typeof b.project_name !== 'string') {
    stderr('cleargate: server returned unexpected response shape.\n');
    exit(7);
  }

  // Save refresh token
  const store = (opts.createStore ?? createTokenStore)();
  const resolved = await (opts.createStore ? opts.createStore() : createTokenStore());
  await resolved.save(opts.profile, b.refresh_token!);

  // Success output (D10)
  stdout(`joined project '${b.project_name}' as '${hostname()}'\n`);
  stdout(`refresh token saved to ${resolved.backend}.\n`);
}
```

- **Commander's positional argument destructuring.** `<invite-url>` angle brackets mean required; `.action((inviteUrl, opts, command))` — `inviteUrl` is first positional, `opts` is the command's own options (empty here), `command` gives access to parent globals via `command.parent!.opts()`. Test R-1 uses lowercase; test R-2 could use mixed-case for coverage.

Test R-5 adjacent: add R-17 "URL with query params stripped" if time permits.
