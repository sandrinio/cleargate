## Locked decisions

### Library pick: `@napi-rs/keyring@^1.2.0`
Verified at planning time via `npm view`:
- Latest stable: `1.2.0` (published 7 months ago, single maintainer, MIT)
- Prebuilt napi binaries published for **all** Node 24 LTS targets named in the SPRINT-03 risk table (L50): `aarch64-apple-darwin`, `x86_64-apple-darwin`, `aarch64-unknown-linux-gnu`, `x86_64-unknown-linux-gnu`. Source: package.json `napi.targets` array in the published tarball.
- Zero runtime dependencies. Engines `>= 10`, compatible with Node 24.
- API: `class Entry` (sync) and `class AsyncEntry`. We use **sync `Entry`** — keychain access is on CLI startup, not a hot path; sync keeps `TokenStore` callers simpler and avoids unnecessary `await`. (We still expose async `TokenStore` methods so a future swap to a network-backed store is non-breaking.)

**Fallback trigger (concrete):**
- If `npm install @napi-rs/keyring@^1.2.0` exits non-zero on Linux x64 in CI (the only target the CI matrix actually runs against in this sprint), Developer swaps the dep for `keytar@^7.9.0` and re-runs. The `KeychainTokenStore` import line is the only file change required (interface is identical in shape — both expose `getPassword/setPassword/deletePassword(service, account)`). No `TokenStore` consumer changes.
- If `npm install` succeeds but `new Entry(...)` throws at first runtime call on Linux CI (e.g. missing `libsecret-1.so.0`), that is **not** a fallback trigger — it's the expected "keychain unavailable" path the factory must handle gracefully (see "Keychain-unavailable detection" below).

### Interface: async, profile-keyed
```ts
// src/auth/token-store.ts
export interface TokenStore {
  /** Persist a refresh token for the named profile. Overwrites any existing value. */
  save(profile: string, token: string): Promise<void>;
  /** Return the stored refresh token for the profile, or null if none. */
  load(profile: string): Promise<string | null>;
  /** Remove the stored token for the profile. Idempotent — no error if absent. */
  remove(profile: string): Promise<void>;
  /** Backend identifier for diagnostics ("keychain" | "file"). */
  readonly backend: 'keychain' | 'file';
}

export interface TokenStoreFactoryOptions {
  /** Override file path for FileTokenStore (test seam). Default: ~/.cleargate/auth.json */
  filePath?: string;
  /** Override keychain service name (test seam). Default: "cleargate". */
  keychainService?: string;
  /** Force a backend, bypassing detection. Used by tests. */
  forceBackend?: 'keychain' | 'file';
  /** stderr writer for the "keychain unavailable" warning. Default: process.stderr.write. */
  warn?: (msg: string) => void;
}

export async function createTokenStore(opts?: TokenStoreFactoryOptions): Promise<TokenStore>;
```

Async because: (a) future swap to async keychain backend or remote store doesn't ripple, (b) file backend's chmod/mkdir IO benefits from `fs/promises` (no blocking on CLI startup once we add a long-lived watch mode in EPIC-001), (c) parallel — it matches `loadConfig`'s sync nature without forcing the same on token IO.

### Auth file shape — multi-profile, single file, plaintext
Path: `~/.cleargate/auth.json` (alongside `config.json` from M1). Mode `0600`. Parent dir `~/.cleargate/` created with `0700` if missing.

```json
{
  "version": 1,
  "profiles": {
    "default":  { "refreshToken": "ct_rt_..." },
    "staging":  { "refreshToken": "ct_rt_..." }
  }
}
```

- **Single file, namespaced by profile.** Sprint constraint (sprint file L80) says `--profile` accepted, default `"default"`. Multi-file (`auth.<profile>.json`) is rejected: harder to enumerate, harder to migrate in EPIC-001's planned profile-switcher UX.
- `version: 1` is a forward-compat hatch. Reader treats unknown `version` as a hard error pointing to "upgrade `@cleargate/cli`".
- `profiles` is the only top-level data field. Unknown profiles are added on first `save`; profiles never collide. Missing profile on `load` returns `null`.
- **Existing-file invariant:** `save("staging", ...)` against an existing `auth.json` containing only `default` must preserve `default`. Read-modify-write, never truncate-and-replace without merge. Validate the read with zod, throw with file path on parse failure (mirrors `loadConfig`'s file-error pattern at `cleargate-cli/src/config.ts:58-67`).
- **Plaintext on disk.** No envelope encryption in M2. Defense is filesystem permissions only. README chunk (added in DoD close-out, L104) flags this limitation: "File-backend stores refresh tokens in plaintext at chmod 600. On a multi-tenant host or shared-home environment, prefer the keychain backend. Future work: OS-level encryption-at-rest."

### Keychain service/account naming
- Keychain service: `"cleargate"` (constant, lowercased). Overridable via `TokenStoreFactoryOptions.keychainService` for tests.
- Keychain account: the profile name verbatim (`"default"`, `"staging"`, etc.).
- Rationale: one entry per profile; `Entry("cleargate", profile)`. macOS Keychain Access shows them grouped under `cleargate`. Collision risk with other apps is near-zero (no other tool registers under `cleargate`); namespace formally if EPIC-001 adds more secret types (`"cleargate.refresh"`, `"cleargate.api"`).

### Keychain-unavailable detection
The factory probes by attempting `new Entry(service, '__cleargate_probe__').getPassword()`. Three outcomes:
1. **Returns `string | null` cleanly** → keychain works → return `KeychainTokenStore`.
2. **`new Entry` constructor throws** (native module load failed, libsecret missing on Linux) → caught → emit single warning to stderr (`'cleargate: OS keychain unavailable, falling back to file storage at <path>. Run with --log-level=debug for details.'`) → return `FileTokenStore`. Probe is invoked once during `createTokenStore`, result cached on the returned store. Never re-probed for the lifetime of the CLI process.
3. **`getPassword()` throws** (e.g. `dbus` not running, prompt cancelled) → same fallback path as (2), with the same warning. Reason: a keychain that can't be read is functionally unavailable for this CLI invocation; falling back to file gives the user a working path. The warning is deliberately short — debug detail (the underlying error message + `.code`) goes to `process.stderr` only when `process.env['CLEARGATE_LOG_LEVEL'] === 'debug'`.

The probe **does not write** to the keychain — `getPassword` on a non-existent entry returns `null` (per `@napi-rs/keyring@1.2.0` `index.d.ts:122`: `getPassword(): string | null`). No cleanup needed.

const ProfileEntrySchema = z.object({ refreshToken: z.string().min(1) }).strict();

**Gotchas surfaced from code inspection:**
- `@napi-rs/keyring@1.2.0` `index.d.ts:122` documents `getPassword(): string | null` (returns `null` for missing), but the doc-comment two lines up says it "Returns a NoEntry error if there isn't one". The actual return type wins, but defensive code must `try/catch` around `getPassword` AND also handle a `null` return — both code paths are real depending on platform and library version. Test #13 covers null; the catch in `KeychainTokenStore.load` covers throw.
- `Entry` is sync — wrapping `Entry.getPassword()` in `Promise.resolve(...)` is fine and keeps the async `TokenStore` interface clean. Do NOT use `AsyncEntry` here — it adds an `AbortSignal` parameter we don't need and forces every test to deal with promise scheduling.
- `fs.writeFile(path, data, { mode: 0o600 })` only sets the mode **on file creation**. If `auth.json` already exists with mode `0644` (e.g. user manually created it), `writeFile` does NOT chmod. Mitigation: after every successful `writeFile`, explicitly call `fs.chmod(path, 0o600)`. Same for the parent dir after `mkdir`.
- Atomic write via `writeFile(tmpPath, ...)` + `rename(tmpPath, finalPath)` is required because `auth.json` may contain other profiles. A crash mid-write must not corrupt the file. Use `path.join(dirname, '.auth.json.tmp')` as the tmp name.
- `os.homedir()` can return `''` on edge containers (per `cleargate-cli/src/config.ts:44-46` we already guard for this in config). FileTokenStore's factory must throw a clear error if both `opts.filePath` is unset and `os.homedir()` is empty — silent fall-through to writing `/.cleargate/auth.json` at filesystem root is unacceptable.
- `fs/promises.mkdir(dir, { recursive: true, mode: 0o700 })` only sets mode on newly created directories. Existing `~/.cleargate/` (created by config.ts user-side) keeps whatever mode it had (likely `0755`). Don't try to "fix" it — it would surprise users who set custom modes. Document in README chunk: "If you want strict mode-700 on `~/.cleargate/`, set it manually."

## Cross-story risks

- **STORY-005-05 (`cleargate join`) will consume this in M4.** It calls `createTokenStore()` → `store.save(profile, refreshToken)`. The `refreshToken` value comes from the MCP `POST /join/:invite_token` JSON response (STORY-003-13). M2's contract is: never log the token, never return it from `save`, accept it as opaque string. STORY-005-05's response-logger strip (sprint risk table L52) is a separate concern — TokenStore must not be the leak vector. The 19 tests above never `console.log` a token value, only assert equality.
- **STORY-005-01..04 (Admin CLI) does NOT use TokenStore.** Per sprint file L49, admin JWT sourcing is `CLEARGATE_ADMIN_TOKEN` env var first, then `~/.cleargate/admin-auth.json` chmod 600. That's a separate file with a separate (likely simpler) loader, deliberately outside `TokenStore` scope. M3 architect must NOT reuse `FileTokenStore` for admin tokens — the JSON shape differs (admin file holds a single JWT, not a profile map). Risk: a M3 developer sees `FileTokenStore` and tries to repurpose it. **Mitigation:** call this out explicitly in M3's architect plan when written.
- **`@napi-rs/keyring` adds a native dep to a package destined for `npx`.** First-time `npx @cleargate/cli` will trigger a postinstall binary download. On networks blocking GitHub releases (where napi prebuilts live) the install will hang then fail. The file-fallback path doesn't help — `npm install` itself fails. README chunk (DoD L104) must document the workaround: `npm config set @napi-rs:registry https://registry.npmjs.org/` (which is the default; the failure mode is more about corporate proxies than registries). No code change required in M2.
- **Existing `auth.json` from a different profile.** If a user has `auth.json` with `{ version: 1, profiles: { default: {...} } }` and later runs `cleargate --profile staging join ...`, `save('staging', ...)` must read+merge+write, not truncate. Test #7 covers exactly this (no-clobber across profiles).
- **Schema strictness blocks forward compatibility.** A v0.2 of the CLI that introduces a `profiles.<name>.deviceId` field would cause v0.1 to throw on `load`. Mitigation today: bump `version: 1 → 2` whenever the per-profile shape changes; v0.1's "wrong version" error message points the user to upgrade. Documented in test #11.

## Risks specific to M2

- **Windows chmod is a no-op.** `fs.chmod(path, 0o600)` on Windows succeeds silently but does nothing (NTFS uses ACLs, not POSIX modes). We do not target Windows in SPRINT-03 (sprint file makes no Windows commitment), but the test for chmod 0600 must `it.skipIf(process.platform === 'win32')` so a Windows contributor's local test run doesn't fail. Same for the `0o700` parent-dir test.
- **CI runs on headless Linux without libsecret** (sprint file L50). The mock-based keychain tests run there fine because `vi.mock('@napi-rs/keyring', ...)` replaces the module before any native code loads. The factory test for "constructor throws → file backend" simulates the real CI condition without needing the real failure. **Acceptance:** `npm test` must pass on Linux x64 with NO `libsecret-1` package installed. Developer verifies by running tests in a fresh container (or just trusts vi.mock — fresh-container verification is a manual gate but not a commit-blocker; it will be exercised when CI runs in a later sprint).
- **Service-name collision with other apps using `@napi-rs/keyring`.** Risk is theoretical (no other tool we control uses `cleargate` as a service name). Mitigation: scoped the service constant in code (`const KEYCHAIN_SERVICE = 'cleargate'`) so a future rename is a one-line change. Account name = profile name keeps the namespace flat per service, which is the keyring-rs idiomatic shape.
- **Probe writes nothing → no leftover state.** Verified above: `getPassword` on a non-existent entry returns `null` per typedef. We never call `setPassword` during probe. No cleanup is required after `createTokenStore`.
- **Atomic rename on Windows.** `fs.rename` on Windows fails if the destination exists in some Node versions. We're not targeting Windows; if we ever do, swap to `fs.cp` + `fs.unlink` or use `node:fs/promises.rename` which on modern Node handles this. No action in M2.
