### D3 ‚Äî Schema sharing: vendor a snapshot of response shapes; reuse zod request schemas via relative import
**Discovery:** the sprint plan L70 referenced `mcp/src/admin-api/schemas.ts` as a single Zod source ‚Äî **that file does not exist** (verified: `ls /Users/ssuladze/Documents/Dev/ClearGate/mcp/src/admin-api/` shows no `schemas.ts`; grep for `schemas` returns only `openapi.ts` and the snapshot). Each handler defines its own `z.object` request body inline (`mcp/src/admin-api/projects.ts:7-9`, `mcp/src/admin-api/members.ts:8-14`, `mcp/src/admin-api/tokens.ts:14-18`). DTOs are TS interfaces, not Zod (`mcp/src/admin-api/projects.ts:15-21`, `members.ts:22-30`, `tokens.ts:25-33`).

**Decision:**
- **Request schemas:** import the inline Zod literals via relative path from `mcp/src/admin-api/{projects,members,tokens}.ts`. The setup commit re-exports them with named exports (`CreateProjectBody`, `InviteBody`, `IssueTokenBody`). This requires a tiny refactor of those three mcp files: `const CreateBody = z.object(...)` ‚Üí `export const CreateProjectBody = z.object(...)`. **Acceptable** ‚Äî it's a rename + export, no logic change, and tightens the shared-source story for EPIC-006.
- **Response schemas:** vendor as Zod parsers in `cleargate-cli/src/admin-api/responses.ts`, hand-authored from `mcp/src/admin-api/__snapshots__/openapi.test.ts.snap` (Project lines 238-271, Member lines 174-223, InviteCreated lines 150-173, TokenIssued lines 345-363, TokenMeta lines 378-424). Rationale: DTOs in mcp are not Zod, so importing types only would forfeit runtime validation. Hand-authored Zod schemas in CLI are ~60 LOC, exactly mirror the OpenAPI snapshot, and run on every API response (defensive). Snapshot drift detection: a unit test in `cleargate-cli/test/admin-api/snapshot-drift.test.ts` reads the openapi snapshot file and asserts each vendored schema's keys match the snapshot's `required` arrays. Test fails loud on drift, prompting CLI bump.

This trades against pure DRY but matches the sprint risk row L51 ("hand-roll a thin AdminApiClient ... import schemas directly OR vendor a snapshot ‚Äî decide at M1"). M3 architect formally chooses the hybrid.

### D4 ‚Äî Admin JWT loader: env first, file second, NOT TokenStore
File path: `~/.cleargate/admin-auth.json`, mode `0600`, parent dir mode `0700`. Shape (zod `.strict()`):
```ts
export const AdminAuthFileSchema = z.object({
  version: z.literal(1),
  token: z.string().min(1),  // a single JWT, NOT a profile map
}).strict();
```
**This is deliberately NOT `FileTokenStore`'s shape.** Per QA-M2 watch item #1 (handed in by orchestrator) and W2.md L205, `FileTokenStore` is a profile‚Üírefresh-token map for the user-facing `cleargate join` flow; the admin file holds a single admin JWT acquired out-of-band via `mcp/scripts/dev-issue-token.ts --role=admin --gh-user=X` (verified at `/Users/ssuladze/Documents/Dev/ClearGate/mcp/scripts/dev-issue-token.ts:79-100`). Reusing `FileTokenStore` would mis-shape both: forcing the admin JWT into a `profiles.default.refreshToken` slot collides with the user's own refresh-token storage on a single-laptop dev workflow.

Loader signature (lives at `cleargate-cli/src/admin-api/admin-auth.ts`):
```ts
export interface LoadAdminAuthOptions {
  env?: NodeJS.ProcessEnv;       // default: process.env
  filePath?: string;             // default: ~/.cleargate/admin-auth.json
  homedir?: () => string;        // test seam, default: os.homedir
}
export interface AdminAuth { token: string; source: 'env' | 'file'; }
export function loadAdminAuth(opts?: LoadAdminAuthOptions): AdminAuth;
// Throws Error with literal message:
//   'No admin token. Set CLEARGATE_ADMIN_TOKEN or write ~/.cleargate/admin-auth.json (chmod 600). See README ¬ßadmin-jwt.'
```
Sync (mirrors `loadConfig` at `cleargate-cli/src/config.ts:33`). Env wins; if `CLEARGATE_ADMIN_TOKEN` is set, file is not read at all (avoids accidentally surfacing a stale file token). File missing ‚Üí not an error if env is set; file malformed ‚Üí always throws (file present means user intended to use it).

**Out of scope:** writing the admin-auth.json file. M3 ships read-only. Users seed the file by hand (one-shot, documented in `mcp/README.md` admin section to be authored at sprint close per DoD L104). No `cleargate-admin login`/`cleargate-admin set-token` subcommand ‚Äî sprint constraint L79.

### D6 ‚Äî Error mapping + exit codes
One error class lives in `cleargate-cli/src/admin-api/errors.ts`:
```ts
export class AdminApiError extends Error {
  constructor(
    public readonly kind: 'network' | 'auth' | 'forbidden' | 'not_found'
                        | 'invalid_request' | 'server' | 'response_shape',
    public readonly status: number | null,
    public readonly details: unknown,
    message: string,
  ) { super(message); }
}
```
HTTP‚Üíkind mapping inside `AdminApiClient`:
- network failure (fetch throws) ‚Üí `kind: 'network'`, `status: null`
- 401 ‚Üí `'auth'` (likely missing/expired JWT)
- 403 ‚Üí `'forbidden'` (JWT valid but not admin role)
- 404 ‚Üí `'not_found'`
- 400 ‚Üí `'invalid_request'` (`details = body.details` when shaped per `Error` schema lines 135-149)
- 5xx ‚Üí `'server'`
- 2xx but Zod parse fails ‚Üí `'response_shape'`

CLI exit-code policy (single shared formatter at `mcp/scripts/commands/_render-error.ts`, used by all four subcommands):
| AdminApiError.kind   | exit | stderr first line                                                 |
|----------------------|------|-------------------------------------------------------------------|
| `network`            | 2    | `cleargate-admin: cannot reach <baseUrl> (<inner>)`                |
| `auth`               | 3    | `cleargate-admin: admin token rejected. Refresh via dev-issue-token --role=admin.` |
| `forbidden`          | 3    | `cleargate-admin: token is not admin-role.`                        |
| `not_found`          | 4    | `cleargate-admin: not found (<resource>).`                         |
| `invalid_request`    | 5    | `cleargate-admin: invalid request: <field>: <issue>` (one line per zod field) |
| `server`             | 6    | `cleargate-admin: server error <status>.`                          |
| `response_shape`     | 7    | `cleargate-admin: server returned unexpected response shape (CLI may be out of date).` |
| Argument parse error | 1    | (Commander default ‚Äî already handled at `cleargate-cli/src/cli.ts:13` `.showHelpAfterError`) |
| Missing admin token  | 3    | message from `loadAdminAuth` (see D4)                              |
| Unhandled exception  | 99   | `cleargate-admin: internal error: <message>`                       |

Exit 0 reserved for success. Codes 2-7 categorize remote failures; tests assert exact codes (matrix in test plan below).

### D7 ‚Äî Plaintext-token strip lives in `AdminApiClient` (not at command layer)
Per the orchestrator's specific scope clarification: `issueToken` returns `TokenIssued` which contains the plaintext `token` field exactly once (server-side `mcp/src/admin-api/tokens.ts:144-148`). Mirroring SPRINT-02's response-logger strip (sprint risk row L52), the CLI's debug logger (any `console.log`/`process.stderr` write that includes the response object) must redact `token` before printing.

Implementation:
- `AdminApiClient` accepts an optional `warn(msg)` (D5). Internally, when in `process.env.CLEARGATE_LOG_LEVEL === 'debug'` mode, the client emits a single `warn('[admin-api] POST /projects/:pid/tokens ‚Üí 201 (token=<redacted>)')` line per request ‚Äî never the raw response object.
- `redactSensitive(obj)` helper at `cleargate-cli/src/admin-api/redact.ts` takes any object, replaces `token` and `refresh_token` and `invite_token` keys (recursive) with the literal string `'<redacted>'`. Used by every debug log path.
- The `issue-token` subcommand's success output (the only place plaintext is printed by design) calls `process.stdout.write(plaintext + '\n')` directly with the bare string ‚Äî never spreads the response object. Test (`Test I-7` below) asserts `process.stderr` never contains the plaintext during a successful `issueToken` call even with `CLEARGATE_LOG_LEVEL=debug`.

The `inviteMember` response also contains an `invite_token` field. Same strip applies. The `invite` subcommand prints `invite_url` (which embeds the token) ‚Äî that's intentional per `STORY-005-02:30` ("stdout contains the invite URL"); only `invite_token` as a bare field is redacted from debug logs.
