## 1. Goal

After M3, `cleargate join` performs the two-step identity-bound redemption end-to-end against the M1+M2 server:

- `cleargate join <invite-url> --auth github` → CLI POSTs `/join/:token/challenge { provider: 'github' }`, displays `verification_uri` + `user_code` from `client_hints`, polls `https://github.com/login/oauth/access_token` directly using the device-flow protocol, then POSTs `/join/:token/complete { challenge_id, proof: { code: <oauth_code> } }`. Member-side mode of `GitHubProvider._completeMemberMode` (`mcp/src/auth/identity/github-provider.ts:261-301`) handles the one-shot exchange.
- `cleargate join <invite-url> --auth email` → CLI POSTs `/challenge { provider: 'email' }`, reads `client_hints.sent_to` (masked email) from the response, prompts the user to type the 6-digit OTP from their inbox, then POSTs `/complete { challenge_id, proof: { code: '123456' } }`.
- `cleargate join <invite-url>` (no `--auth`) → interactive picker (TTY) or auto-pin to invite's `required_provider` (when server returns `provider_not_allowed` for the wrong choice). `--non-interactive` without `--auth` exits 1.
- `cleargate admin login` UX is **byte-identical** to today — `cleargate-cli/test/commands/admin-login.snapshot.test.ts` (already shipped in CR-004) stays green unchanged. Refactor extracts the device-flow polling loop from `admin-login.ts:151-216` into `identity-flow.startDeviceFlow(...)` so `join --auth github` reuses it.

`refresh_token` from `/complete` is saved via the existing `TokenStore.save()` path (`cleargate-cli/src/commands/join.ts:147-148`) — that helper is unchanged.

## 3. Wire-shape contracts (server-side, frozen post-M2)

### `POST /join/:invite_token/challenge` (verified `mcp/src/routes/join.ts:88-185`)

Request: `{ "provider": "github" | "email" }`

Response 200:
```json
{
  "challenge_id": "<uuid-v4>",
  "provider": "github" | "email",
  "expires_in": 600,
  "client_hints": { ... }
}
```

`client_hints` for GitHub (per `github-provider.ts:173-179`): `{ device_code, user_code, verification_uri, expires_in, interval }` — exact fields M3 CLI must read.

`client_hints` for email (per `magic-link-provider.ts` — verified line 100 `clientHints: { sent_to: maskEmail(expectedEmail) }`): `{ sent_to: 'a***@example.com' }` — single field.

Errors per M1 §4: 400 `invalid_request` | `provider_not_allowed` | `provider_unknown`; 404 `not_found`; 410 `invite_already_consumed` | `invite_expired`; 429 (rate-limit).

### `POST /join/:invite_token/complete` (verified `mcp/src/routes/join.ts:188-337`)

Request: `{ "challenge_id": "<uuid>", "proof": <provider-shaped> }`

For email provider: `proof: { "code": "123456" }` (per `magic-link-provider.ts:88-92`).

For GitHub provider (member-side): `proof: { "code": "<oauth_code_or_access_token>" }` (per `github-provider.ts:189-194` — dispatch by `'code' in proof`). **OD-2** flags the access_token-vs-authorization_code semantics question.

Response 200 — byte-identical to legacy single-shot body:
```json
{
  "refresh_token": "...",
  "refresh_token_expires_in": 2592000,
  "project_id": "...",
  "project_name": "...",
  "member_id": "...",
  "member_role": "user"
}
```

Errors per M1 §4: 400 `invalid_request` | `identity_proof_required`; 403 `email_mismatch`; 410 `challenge_expired` | `invite_already_consumed` | `invite_expired`; 502 `provider_error`.

**OD-2: GitHub member-side proof semantics.** `mcp/src/auth/identity/github-provider.ts:271-281` (member mode) POSTs to GitHub with `grant_type: 'authorization_code'` + `code`.
