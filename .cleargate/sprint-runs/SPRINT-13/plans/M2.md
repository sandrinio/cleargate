## 3. IdentityProvider contract reminder

Both providers MUST implement the M1 §5 interface verbatim (`mcp/src/auth/identity/provider.ts:29-33`):

```ts
export interface IdentityProvider {
  readonly name: string;  // 'github' | 'email'
  startChallenge(input: { inviteId: string; expectedEmail: string }): Promise<{
    payload: Record<string, unknown>;       // jsonb stored in identity_proofs.challenge_payload — provider-private
    clientHints: Record<string, unknown>;   // public-facing; returned in /challenge response body
  }>;
  completeChallenge(input: {
    challengeId: string;
    payload: Record<string, unknown>;       // round-tripped from startChallenge
    proof: unknown;                          // client-supplied (OAuth code | OTP | etc.)
  }): Promise<{ verifiedEmail: string; providerSubject: string }>;
}
```

Provider throws on failure → route at `routes/join.ts:238-255` maps to 502 `provider_error`. Provider returns mismatched email → route maps to 403 `email_mismatch` at `routes/join.ts:258-270`.

This works for the admin-login path which uses its OWN endpoints (`/auth/device/start` + `/auth/device/poll`), NOT `/join/*`. **The member-side path (`/join/*`) is GitHub-Provider-via-proof: the CLI does the GitHub polling itself and only calls `/join/:token/complete` ONCE with the final OAuth `code`.** This requires that `GitHubProvider.completeChallenge` accept TWO modes: (a) admin-login mode — payload has `device_code`, proof is unused, provider polls GitHub itself; (b) member-join mode — payload has the device_code, proof has `{code: oauth_code}` already-exchanged by the CLI. **Decision: keep ONE mode = admin-login mode for M2 (the CLI in M3 will pass the device_code through the same payload contract).** The member-side CLI in M3 will: call `/challenge` → gets `client_hints.device_code/user_code/verification_uri`, displays them, then loops calling `/complete` with empty proof until 200 returns. This matches the admin-login UX shape per EPIC-019 §6 Q4 ("paste-6-digit-code in CLI" — for GitHub it's "open URL + enter user_code in browser, CLI polls"). This means M2 CR-004 ships `GitHubProvider` with poll-on-each-call semantics; the route returns 502 `provider_error` while pending. **Open Decision §10.2:** the route returning 502-while-pending is semantically wrong — pending is not an error. Pure M2 either accepts the 502-while-pending wire shape (CLI in M3 maps it to "keep polling") OR M2 adds a non-error `pending` result type to the IdentityProvider interface (substrate diff = M1 modification = §11 guardrail violation). ## 10. Open decisions for orchestrator

## 10a. Orchestrator resolutions (2026-04-25)

1. **TTL+rate-limit-only — ACCEPTED.** No literal 5-attempt counter in M2. Justification: 6-digit OTP space (1M) × 10-min TTL × anonymous rate-limit (10 req/15min on `/join/*` per `routes/join.ts:82`) caps brute-force at ~7 attempts/TTL → ~7/1M ≈ 7e-6 success probability per challenge. Defence-in-depth, not security necessity. M1 substrate stays frozen. Reporter logs the deviation from CR-005 §3 in REPORT.md.
2. **Two-mode `completeChallenge` dispatched by proof shape — single GitHubProvider, no substrate change.**
   - **Admin-login mode (server polls):** caller passes `proof: undefined` (or absent). Provider polls GitHub `/login/oauth/access_token` once per call. On `authorization_pending`/`slow_down` → throws `GitHubChallengeError('pending'|'slow_down', {interval?})`. The `auth-device-poll.ts` route handler catches and returns 200 `{pending: true, retry_after}` — preserving current CLI wire shape (snapshot-test gated).
   - **Member-side mode (client passes already-exchanged code):** caller passes `proof: {code: <oauth_code>}`. Provider performs one-shot access_token exchange + `/user/emails` fetch. No polling, no pending. Member-side `/join/:token/complete` therefore never returns 502-while-pending — only on real errors.
   - M3 CR-006 CLI for `cleargate join --auth github` does the device-flow polling client-side (calls GitHub directly using `client_hints.device_code` from `/challenge` response), then POSTs `/complete` once with `proof: {code}`. This matches EPIC-019 §6 Q4 (paste-equivalent UX).
   - Document the two modes inline in `github-provider.ts` with a doc-comment block.
   - **CR-004 dev:** implement both branches in `completeChallenge`; cover both with separate tests (admin-login mode = GP-04..GP-08 with `proof: undefined`; member-side mode = new GP-10..GP-13 with `proof: {code}` happy/error paths). Update §2 GP-* test list accordingly.
3. **Snapshot test stays in `cleargate-cli/test/commands/admin-login.snapshot.test.ts`** (architect's pick — CONFIRMED). CLI stdout bytes are the user-visible regression contract per R-03. Server-side wire-shape is implicit-tested by the existing DP-1..DP-9 suite already.
