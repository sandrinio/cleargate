```ts
export const identityProofs = pgTable(
  'identity_proofs',
  {
    id: uuid().primaryKey().defaultRandom(),
    inviteId: uuid()
      .notNull()
      .references(() => invites.id, { onDelete: 'cascade' }),
    provider: text().notNull(), // 'github' | 'email' | future
    expectedEmail: text().notNull(), // snapshot of members.email at challenge-start
    verifiedEmail: text(), // NULL until /complete; set to provider-attested address
    providerSubject: text(), // NULL until /complete; provider's stable user id
    challengePayload: jsonb().notNull().default(sql`'{}'::jsonb`), // opaque; provider-private (PKCE verifier hash, nonce hash, etc.) — NEVER store plaintext nonces here (FLASHCARD #plaintext-redact)
    createdAt: timestamp({ withTimezone: true }).defaultNow().notNull(),
    expiresAt: timestamp({ withTimezone: true }).notNull(), // ≤10min
    consumedAt: timestamp({ withTimezone: true }), // NULL = pending; one-shot
  },
  (t) => [
    index('idx_identity_proofs_invite').on(t.inviteId),
    index('idx_identity_proofs_pending')
      .on(t.expiresAt)
      .where(sql`${t.consumedAt} IS NULL`),
  ],
);
export type IdentityProof = typeof identityProofs.$inferSelect;
```

Also add `export type NewIdentityProof = typeof identityProofs.$inferInsert;` in the Type helpers section.

## 4. Endpoint surface

Both routes mount inside the existing scoped register block in `routes/join.ts` (L38) so they share the `anonLimit` preHandler.

### `POST /join/:invite_token/challenge`

Request body:
```json
{ "provider": "github" | "email" | "fake" }
```

Response 200:
```json
{
  "challenge_id": "uuid-v4",
  "provider": "github",
  "expires_in": 600,
  "client_hints": { /* provider-opaque, e.g. { "device_code_url": "..." } for github; for fake: { "secret": "ABC123" } */ }
}
```

Errors:
- 400 `invalid_request` — body fails Zod parse
- 400 `provider_not_allowed` — invite's `required_provider` is set and `≠` requested provider
- 400 `provider_unknown` — provider name not in registry
- 404 `not_found` — invite UUID malformed or row missing
- 410 `invite_already_consumed` — invite.consumed_at IS NOT NULL
- 410 `invite_expired` — invite.expires_at ≤ now()

Side effects: insert one `identity_proofs` row with `expectedEmail = invite.email`, `expiresAt = now() + 10min`, `consumedAt = NULL`. Audit-log: `tool='invite_challenge_start'`, `result='ok'|'error'`, `errorCode=<code>` on error. Payload column receives provider-returned opaque state (see §5 — never plaintext secrets).

### `POST /join/:invite_token/complete`

Request body:
```json
{ "challenge_id": "uuid-v4", "proof": { /* provider-shaped */ } }
```

Response 200 — **byte-identical to current `/join/:token` 200 body** (preserve contract):
```json
{
  "refresh_token": "...",
  "refresh_token_expires_in": 2592000,
  "project_id": "...",
  "project_name": "...",
  "member_id": "...",
  "member_role": "user"
}
```

Errors:
- 400 `invalid_request` — body fails Zod parse
- 400 `identity_proof_required` — challenge_id missing or not found for this invite
- 403 `email_mismatch` — provider-verified email ≠ identity_proofs.expectedEmail (case-insensitive compare). Invite + identity_proof both remain unconsumed.
- 410 `challenge_expired` — identity_proofs.expires_at ≤ now() OR consumed_at IS NOT NULL
- 410 `invite_already_consumed` / `invite_expired` — invite re-checked atomically (race-safe)
- 502 `provider_error` — provider.completeChallenge throws (network, GitHub down, etc.)

Side effects (single transaction):
1. Call `provider.completeChallenge({ challengeId, proof })` → `{ verifiedEmail, providerSubject }`.
2. `BEGIN` transaction: lock identity_proofs row `FOR UPDATE`; verify pending + unexpired; verify `verifiedEmail.toLowerCase() === expectedEmail.toLowerCase()`; UPDATE identity_proofs set `verifiedEmail`, `providerSubject`, `consumedAt = now()` WHERE id = challenge_id AND consumed_at IS NULL; same atomic UPDATE on invites set `consumed_at = now()` WHERE id = invite_id AND consumed_at IS NULL AND expires_at > now() RETURNING member_id, project_id, role; if 0 rows updated → 410. Mint refresh as today (join.ts L90).
3. Audit-log: `tool='invite_challenge_complete'`, `result='ok'`. On reject: `tool='invite_redeem_reject'`, `errorCode='email_mismatch'|'challenge_expired'|...`. **Never** include the proof, verifiedEmail, providerSubject, or challenge_id in audit payload (FLASHCARD #plaintext-redact). The `audit_log` table has no jsonb payload column anyway (verified `mcp/src/db/schema.ts:134-153`); only `tool`, `result`, `errorCode` are written — safe by construction.

### Residual `POST /join/:invite_token`

Returns 400 `{ "error": "identity_proof_required", "hint": "POST to /join/:token/challenge first" }` unconditionally. Audit-log: `tool='invite_redeem_reject'`, `errorCode='identity_proof_required'`. Required by EPIC-019 §5 Gherkin "Bearer-only redemption is refused" + DoD bullet 6.

## 5. IdentityProvider interface

`mcp/src/auth/identity/provider.ts`:

```ts
export interface ChallengeStartInput {
  inviteId: string;
  expectedEmail: string;
}

export interface ChallengeStartResult {
  /** Opaque state the provider needs at /complete time. Persisted in identity_proofs.challenge_payload as jsonb.
   *  MUST NOT contain plaintext nonces, OAuth codes, or magic-link codes — store hashes/derived state only. */
  payload: Record<string, unknown>;
  /** Public-facing hints sent back to the client (e.g. device-code URL, "check your inbox"). */
  clientHints: Record<string, unknown>;
}

export interface ChallengeCompleteInput {
  challengeId: string;
  /** Provider-private storage from startChallenge. */
  payload: Record<string, unknown>;
  /** Client-supplied proof: OAuth code for github, 6-digit OTP for email, free-form for fake. */
  proof: unknown;
}

export interface ChallengeCompleteResult {
  /** Provider-attested verified email. /complete handler compares to expectedEmail. */
  verifiedEmail: string;
  /** Stable provider-side identity (e.g. github user_id). Persisted in identity_proofs.provider_subject. */
  providerSubject: string;
}

export interface IdentityProvider {
  readonly name: string;
  startChallenge(input: ChallengeStartInput): Promise<ChallengeStartResult>;
  completeChallenge(input: ChallengeCompleteInput): Promise<ChallengeCompleteResult>;
}
```

`mcp/src/auth/identity/registry.ts`:

```ts
import type { IdentityProvider } from './provider.js';

export class ProviderRegistry {
  private map = new Map<string, IdentityProvider>();
  register(p: IdentityProvider): void {
    if (this.map.has(p.name)) throw new Error(`duplicate provider: ${p.name}`);
    this.map.set(p.name, p);
  }
  get(name: string): IdentityProvider | undefined { return this.map.get(name); }
  list(): string[] { return [...this.map.keys()]; }
  has(name: string): boolean { return this.map.has(name); }
}
```

Wiring: build a singleton `ProviderRegistry` at app boot in the same file that calls `registerJoinRoute` (one level up — find the existing app builder; do NOT instantiate inside `routes/join.ts`). Pass it as a new field on `JoinRouteDeps`. In tests, register the FakeProvider before `app.ready()`. In CR-003 production wiring, register NOTHING by default — M2 adds GitHub + email registrations. This means production deploys of CR-003-only would 400 every challenge with `provider_unknown`; that is intentional (the CR-003 substrate is not user-facing on its own; CR-004/CR-005 light it up).

`mcp/src/auth/identity/fake-provider.ts`:

```ts
import { randomBytes } from 'node:crypto';
import type { IdentityProvider, ChallengeStartInput, ChallengeStartResult, ChallengeCompleteInput, ChallengeCompleteResult } from './provider.js';

/** Test-only. Round-trips a 6-char secret. Returns the verified email = expected email
 *  when proof.secret matches; throws Error('email_mismatch_simulation') when caller sets
 *  proof.simulateMismatch=true so the route can exercise the 403 path.
 *  NEVER register this in production wiring. */
export class FakeProvider implements IdentityProvider {
  readonly name = 'fake';

  async startChallenge(input: ChallengeStartInput): Promise<ChallengeStartResult> {
    const secret = randomBytes(3).toString('hex'); // 6 chars
    return {
      payload: { secretHash: this.hash(secret), expectedEmail: input.expectedEmail },
      clientHints: { secret }, // tests read this; real providers would NOT echo a secret
    };
  }

  async completeChallenge(input: ChallengeCompleteInput): Promise<ChallengeCompleteResult> {
    const proof = input.proof as { secret?: string; simulateMismatch?: boolean };
    const expectedEmail = input.payload.expectedEmail as string;
    if (proof.simulateMismatch) {
      return { verifiedEmail: 'mallory@evil.com', providerSubject: 'fake-mallory' };
    }
    if (!proof.secret || this.hash(proof.secret) !== input.payload.secretHash) {
      throw new Error('fake_proof_invalid');
    }
    return { verifiedEmail: expectedEmail, providerSubject: 'fake-' + expectedEmail };
  }

  private hash(s: string): string {
    return require('node:crypto').createHash('sha256').update(s).digest('hex');
  }
}
```

Note: `clientHints.secret` echoes the secret only for the FakeProvider — that is the test seam. Real providers NEVER echo secrets in clientHints.
