### Single-flight cache design (authoritative)

- **Storage:** module-level `Map<string, { accessToken: string; expiresAtMs: number }>` inside `cleargate-cli/src/auth/acquire.ts`. Process-local — naturally cleared when the Node CLI exits. Test seam: exported `__resetAcquireCache()` called in `beforeEach`.
- **Key (Risk R1):** `` `${profile}::${mcpUrl}` ``. Profile is mandatory in the key; two CLI runs in the same Node process against two different profiles MUST hit `/auth/refresh` twice. Unit test for this is non-negotiable per R1 mitigation row.
- **Expiry rule:** `expiresAtMs = (access_token.payload.exp - 60) * 1000`. Cache hit requires `Date.now() < expiresAtMs`. The 60-second buffer prevents race-expiry where a long-running multi-call operation (sync's list + N pulls + pushes) sees a token age past `exp` mid-loop.
- **`--refresh` bypass:** `opts.forceRefresh === true` skips the cache read AND overwrites the cache entry with the fresh token. Plumbed only from `conflicts --refresh` in this story; other commands never set it.
- **Env short-circuit never caches:** `CLEARGATE_MCP_TOKEN` path returns immediately without touching the Map. Two reasons: (a) env tokens have no known `exp` without decoding; (b) CI sets env tokens per-invocation anyway.
- **No `exp` claim:** if JWT decode fails or `exp` is missing/non-numeric, do NOT cache. Next call refreshes again. Safe-fail, not safe-cache.

### Chain order + wiring site

**Registration site:** `mcp/src/mcp/transport.ts:31-38`. The `/mcp` route registers `preHandler: [requireAuth, rateLimit]`. After this story the chain becomes `preHandler: [combinedAuth, rateLimit]` where `combinedAuth` is produced by a new factory `buildCombinedAuth(jwt, db)` in `middleware.ts`.

**Sequence inside `combinedAuth`:**

1. Read `request.headers.authorization`. If missing or not `Bearer `: reply 401 `{error: 'missing_token'}`. (Same as today's `middleware.ts:13-15`.)
2. Extract `token = header.slice('Bearer '.length).trim()`.
3. **Try JWT path first.** `try { request.claims = await jwt.verifyAccess(token); return; } catch (err) { /* fall through */ }`. On success: no service-token lookup, return early. Matches current `middleware.ts:17-22`.
4. **Try service-token path.** Call into `buildServiceTokenAuth(jwt, db)(request)` — this helper either sets `request.claims` and returns, or returns without setting claims (no match / bcrypt miss).
5. **If `request.claims` still unset:** reply 401 `{error: 'invalid_token'}`.

**`buildRequireAuth` signature change:** Do NOT modify `buildRequireAuth` — instead add a new exported factory `buildCombinedAuth(jwt, db)` in `middleware.ts` that wraps the full sequence. `buildRequireAuth` stays as-is so admin-api routes (which register it separately) keep their JWT-only behaviour — the story explicitly forbids service-token auth on `/admin-api/v1/*` (§1.2).

### `service-token.ts` (new file) contract

```ts
export function buildServiceTokenAuth(jwt: JwtService, db: DB): (req: FastifyRequest) => Promise<void>
```

Body flow:

1. Read `token` from `Authorization: Bearer <token>` (caller may already have extracted it; prefer re-extraction here so the helper is independently testable).
2. **Shape guard (Risk R2):** regex `^cg_[A-Za-z0-9_-]{32,}$`. **WARNING — discrepancy with existing plaintext format:** `mcp/src/admin-api/tokens.ts:47-49` generates plaintexts as `randomBytes(32).toString('base64url')` (43 chars, no `cg_` prefix). The story (§1.2) and sprint risk R2 both assume a `cg_`-prefixed format. **This is an open decision — see "Open questions" below.** Interim behaviour: shape guard is `^[A-Za-z0-9_-]{40,}$` (matches existing 43-char base64url plaintext, still rejects JWT-shaped strings because JWTs contain `.`). Developer MUST confirm with orchestrator before coding.
3. `looksLikeJwt(s)` helper: `s.split('.').length === 3`. If true: return without setting claims — outer chain already tried JWT and failed, fall through to 401.
4. **Candidate query:**
   ```sql
   SELECT t.id, t.token_hash, t.member_id, m.project_id, m.role
   FROM tokens t
   INNER JOIN members m ON m.id = t.member_id
   WHERE t.revoked_at IS NULL
     AND (t.expires_at IS NULL OR t.expires_at > NOW())
   ```
   Scope is project-scoped in practice (the SELECT returns all active tokens cluster-wide — acceptable because bcrypt eliminates false matches and the set is small). **Optional optimisation for later:** if the plaintext encodes a token-id prefix, restrict. Not in this story.
5. **Timing-flattened bcrypt loop (Risk R3):** Always run ≥1 `bcrypt.compare`. If the candidate query returns 0 rows, run `bcrypt.compare(token, FIXTURE_HASH)` once and discard the result. `FIXTURE_HASH` = a module-level constant, a known bcrypt hash of a random string (pre-computed once at module load via `bcrypt.hashSync('__not_a_real_token__', 12)` — safe: the plaintext is public, the hash is just for timing symmetry). Loop body:
   ```ts
   let matched: Row | null = null;
   if (rows.length === 0) {
     await bcrypt.compare(token, FIXTURE_HASH); // discard
   } else {
     for (const row of rows) {
       const ok = await bcrypt.compare(token, row.token_hash);
       if (ok) { matched = row; break; }  // break OK — attacker can't distinguish 1-compare from N-compares at the network level against p99 bounds
     }
   }
   if (!matched) return; // caller returns 401
   ```
6. **Redis revocation double-check:** before accepting the match, `redis.get(\`rev:token:${matched.id}\`)`. If present, treat as no-match. (Belt-and-braces with the `revoked_at IS NULL` filter — `mcp/src/admin-api/tokens.ts:181-186` writes this key on revoke.) **Note:** `db` alone is not enough — factory signature must be `buildServiceTokenAuth(jwt, db, redis)`. Add `redis` to call sites.
7. **Update `last_used_at`:** fire-and-forget `UPDATE tokens SET last_used_at = NOW() WHERE id = $1`. Await it to keep test determinism (the Gherkin "tokens.last_used_at was updated" scenario asserts this post-call).
8. **Set `request.claims`:** construct `AccessClaims` shape matching `mcp/src/auth/jwt.ts:14-16`:
   ```ts
   request.claims = {
     sub: matched.member_id,
     project_id: matched.project_id,
     role: matched.role,  // 'user' | 'service' | 'admin' — cast via schema enum
     client_id: 'service-token',
     type: 'access',
     // iat/exp omitted — consumers of request.claims (e.g. transport.ts:40-49) only read sub/project_id/role/client_id.
   };
   ```
9. **No plaintext logging:** do NOT call `request.log.info({ token, matched })`. If you need to log at all, log `{ token_id: matched.id, member_id: matched.member_id }` and nothing else. Pino's existing redact at `mcp/src/server.ts:18-28` covers `*.token` + `req.headers.authorization` — do not rely on it; reach for named fields only.

### SQL sequence (transactional)

Per story §1.2. Use a single `BEGIN; ... COMMIT;` block. `gen_random_uuid()` is generated by Postgres — no `uuid` npm dep in CLI. (`admin_users.id` default at `mcp/src/db/schema.ts:21` is `uuid().primaryKey().defaultRandom()` which resolves to `gen_random_uuid()` in the migration — rely on the default, or call it explicitly.)

```
BEGIN;

-- Step 1: does the handle already exist?
SELECT id, is_root FROM admin_users WHERE github_handle = $1;

-- Branch A: row exists + is_root=true  → "already exists; no change." exit 0
-- Branch B: row exists + is_root=false + --force  → UPDATE admin_users SET is_root=true WHERE github_handle=$1 → "Promoted …" exit 0
-- Branch C: row exists + is_root=false + no --force → "use --force to promote" exit non-zero
-- Branch D: no row + another root exists + no --force:
     SELECT count(*) FROM admin_users WHERE is_root = true;
     -- if >= 1: stderr "refusing to create a second root admin; pass --force to override" exit non-zero
-- Branch E: no row + (first root OR --force):
     INSERT INTO admin_users (github_handle, is_root) VALUES ($1, true)
       ON CONFLICT (github_handle) DO UPDATE SET is_root = EXCLUDED.is_root;
     -- "Bootstrapped root admin '<handle>'." exit 0

COMMIT;
```

- Let `gen_random_uuid()` supply the `id` via the schema default — omit the column from INSERT.
- `created_at` likewise defaults to `NOW()` — omit.
- `createdBy` (`schema.ts:29`) is nullable, leave NULL for the bootstrap row. The self-referential FK is nullable explicitly for this case.
- `ON CONFLICT (github_handle) DO UPDATE SET is_root = EXCLUDED.is_root` makes the INSERT re-run idempotent when Branch A would have caught it but a race inserted the row between the SELECT and the INSERT. Cheap belt-and-braces.

### Handle regex + password scrubber

  ```ts
  function scrub(s: string): string { return s.replace(SCRUB, '$1:***@'); }
  ```
  Apply to `err.message` in connection-error catch AND also to any stringified `err` (pg sometimes puts the URL in `err.stack`). Scrub both.

### Connection lifecycle

- `const client = (opts.pgClientFactory ?? ((url) => new pg.Client({ connectionString: url })))(url);`
- `try { await client.connect(); await runSql(client); } catch (err) { stderr(`cleargate: error: cannot reach database (${scrub(err.message)})\n`); exit(3); return; } finally { try { await client.end(); } catch { /* ignore */ } }`.
- Exit codes: 0 success (incl. idempotent no-op); 1 validation error (bad handle, already-have-a-root-no-force); 2 missing DATABASE_URL; 3 connection failure. Keep consistent with `admin login` patterns; note `admin-login.ts:116` uses `process.exit(code)` default — mirror.
