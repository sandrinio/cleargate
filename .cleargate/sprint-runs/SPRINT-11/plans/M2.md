**Files to modify:**

  1. Reads `path.join(cwd, '.cleargate', 'wiki', 'index.md')` byte length. If file missing, skip (no finding — the wiki may not have been built yet; other checks will fire).
  2. Loads ceiling via `loadWikiConfig(cwd).wiki.index_token_ceiling`.
  3. `tokens = Math.round(bytes / 4)`.
  4. If `tokens > ceiling`, push a finding with line `index-budget: wiki/index.md exceeds token ceiling: ${tokens} > ${ceiling}. Shard or prune (see EPIC-015).` (category string literal = `index-budget:` — novel 9th category; extend the contract test at `lint.test.ts` L547-751 to include it).
  5. `--suggest` path: emit `stdout("index token usage: ${tokens} / ${ceiling} (${pct}%)\n")` **unconditionally** before the advisory loop. Over-ceiling stays non-finding in suggest mode (existing Scenario 9 invariant).
- `cleargate-cli/src/commands/wiki-lint.ts` `WikiLintOptions` — no new fields required. `cwd` seam at L48 already threads through fixture paths, so tests can drop `.cleargate/config.yml` + `.cleargate/wiki/index.md` into tmpdir and assert outputs.

**Schema changes:** None. `.cleargate/config.yml` is a new optional file; no existing schema to migrate.

**Test scenarios (from §2.1 Gherkin, each maps to one `it` block):**
1. **Within ceiling passes silently.** Write `.cleargate/wiki/index.md` of exactly 12000 chars (→3000 tokens); ceiling defaults to 8000. Assert `exitCode === 0`, stdout does NOT contain `index-budget:`.
2. **Over ceiling fails.** Write 36000-char index (→9000 tokens); default ceiling 8000. Assert `exitCode === 1`, stdout contains `index-budget: wiki/index.md exceeds token ceiling: 9000 > 8000`, stdout contains `lint: FAIL`.
3. **Custom ceiling from config.** Write `.cleargate/config.yml` with `wiki:\n  index_token_ceiling: 4000`; write 20000-char index (→5000 tokens). Assert `exitCode === 1`, stdout references `5000 > 4000`.
4. **`--suggest` never fails.** Write 36000-char index; mode: 'suggest'. Assert `exitCode === 0`, stdout contains `index token usage: 9000 / 8000 (113%)`, findings prefixed `[advisory]`.

Suggest-mode percentage formula: `Math.round((tokens / ceiling) * 100)`. Test scenario 4 locks the literal `113%`.

**Reuse (no duplication):**
- `loadWikiPages`, `checkOrphan`, etc. from `../wiki/lint-checks.js` — untouched.
- `yaml` from `js-yaml` — already a dep (see `cleargate-cli/package.json` L45).
- Do NOT add a new field to `WikiPage` schema (flashcard `2026-04-19 #wiki #schema #lint`); the ceiling lives in config, not page frontmatter.

**Ordering inside `sprintArchiveHandler` (critical):** The current handler stamps + moves each plan entry in a single loop (L397-408), then truncates `.active` (L412), then runs git. The new sequence:
```
1. stampSprintClose(sprintFile)                           → snapshot captured
2. try { wikiBuildFn(); wikiLintFn(); }
   catch (e) { atomicWriteStr(sprintFile, previousContent); exit 1 }
3. plan-loop stamp + move (unchanged; L397-408)
4. truncate .active (unchanged)
5. git checkout / merge / branch -d (unchanged)
```
Rationale: stamping + building + linting must all succeed *before* any file rename. If lint fails, the raw sprint file must still live at its pending-sync path — moving it first then rolling back the frontmatter leaves the file in archive/ with a Planned status (worst of both worlds).
