## 2. Architectural integrity — mechanism choice

**Choice: SKIP_FILES (correct).**

Three candidate mechanisms existed:

1. **SKIP_FILES** — "don't copy at all" (chosen).
2. **FIRST_INSTALL_ONLY** — "copy on fresh install but never overwrite on re-init."
3. **Path-rewrite** — modify `payloadDir` walk root or filter.

**Why SKIP_FILES is right and FIRST_INSTALL_ONLY is wrong:**

`MANIFEST.json` is a meta-repo build artefact generated by `npm run prebuild` to verify the npm-payload byte-integrity. It has **no semantic role** in a user-repo. FIRST_INSTALL_ONLY would still copy it on fresh installs (the very case where users adopt ClearGate), polluting first-time adopters' repo roots — defeating the bug fix. SKIP_FILES correctly says "this file never belongs at user-cwd, regardless of install state."

**Why path-rewrite is wrong:** `MANIFEST.json` lives at the canonical `cleargate-planning/MANIFEST.json` path (verified tracked). The bug isn't path drift; it's that the npm payload's structural root (`cleargate-cli/templates/cleargate-planning/`) places `MANIFEST.json` directly under `payloadDir` so the recursive walk picks it up. Skipping by relative-path key (`'MANIFEST.json'`) is the targeted fix; rewriting walk roots would risk skipping legitimate `.cleargate/`-rooted content.

**JSDoc clarity:** Dev added a SKIP_FILES bullet at L49-53 explaining the rationale, mirroring the existing `CLAUDE.md` entry's pattern (BUG-016 style). Future readers can disambiguate SKIP_FILES vs FIRST_INSTALL_ONLY without re-deriving it. No surprising behavior introduced.

**No regression to legitimate `.cleargate/`-rooted files:** the SKIP_FILES check applies only to files matching the relative-path key; subdirectory `MANIFEST.json` files (none currently in payload, but defensive against future) would still copy. The QA-Red sanity scenario explicitly verifies `cleargate-planning/CLAUDE.md` and `cleargate-planning/.cleargate/FLASHCARD.md` still copy through.
