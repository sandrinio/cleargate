The self-repair integration MUST happen at L106 (post success-gate, pre `# On failure` banner block at L107). This guarantees: (a) success path (L103–L105) is untouched; (b) the captured `STDOUT_TMP` and `STDERR_TMP` files are still readable from disk for pattern-match dispatch (the trap at L86 only fires on EXIT, not yet); (c) on retry-success, the wrapper proceeds with EXIT_CODE=0-equivalent semantics — Dev decides whether to overwrite EXIT_CODE in-place OR `exit 0` from the sibling. Recommended pattern: sibling exits with the post-retry exit code, parent script reads it via `EXIT_CODE=$?`, then the existing L107+ block writes the incident file (with the new `retry_attempt` field populated). This preserves observability per CR-057 §1 New Logic point 2.

**Sibling contract (if shipped):**
- Inputs (env): `RUN_SCRIPT_NO_REPAIR` (bypass flag, "1" = skip), the retained `STDOUT_TMP` + `STDERR_TMP` paths (sibling reads them), `EXIT_CODE`, original argv (re-exec target).
- Output: writes new exit code to a tmp marker file OR exits with the retried exit code; writes a `retry_attempt: 1` marker the parent reads when constructing the incident JSON.
- Hard caps: max 3 patterns (matched in declared priority order), max 1 retry per pattern, max 60s wall-clock total (single retry × at most 60s sibling-budget).

**Bounce risks (CODE-MODE):**

- **R1: Sibling fork-bomb via `RUN_SCRIPT_ACTIVE` not propagated.** The wrapper sets `RUN_SCRIPT_ACTIVE=1` (live L89) before invoking the wrapped command. The retry MUST keep `RUN_SCRIPT_ACTIVE=1` exported when re-executing — otherwise on retry the wrapped command, if it itself shells out via `run_script.sh`, would re-enter the capture path. **Dev MUST verify** the sibling preserves `export RUN_SCRIPT_ACTIVE=1` across the retry exec.
- **R2: TRAP cleanup races the sibling.** `trap 'rm -f "$STDOUT_TMP" "$STDERR_TMP"' EXIT` at L86 fires on shell exit. If the sibling spawns a subshell that exits before the parent reads the retry result, the tmp files survive. If the parent's outer script-end fires while the sibling is mid-retry, the trap deletes the live capture files. **Mitigation:** sibling must complete synchronously before the parent's L107+ block runs; do NOT background the sibling.
- **R3: Schema-evolution orphan (FLASHCARD 2026-05-04 #cr-046 #wrapper #breaking-change).** `ScriptIncident.retry_attempt: number` added as optional (`number | undefined`) — older incident JSON files lack the field; `isScriptIncident` MUST treat it as optional. Dev verifies by parsing one of the 3 existing SPRINT-24 incident files through the updated type guard.
- **R4: 60s budget cap not enforced.** "max 60s total wall-clock" needs a `timeout` wrapper or explicit `SECONDS=0; … (( SECONDS < 60 ))` check. Bare `bash` retry without a budget cap can hang indefinitely on the very pattern self-repair is targeting. Dev MUST implement an actual wall-clock guard, not just a code comment claiming the cap.
- **R5: Stderr signature drift across CR-054.** CR-054 changed the stderr-truncation byte-count semantics; signatures captured pre-CR-054 may have char-truncated tails. Dev MUST construct signatures against POST-CR-054 stderr captures (i.e. re-running the failing command in the post-CR-054 wrapper produces the canonical signature for matching). Pre-CR-054 incident JSON entries are usable for tally-frequency only, not for verbatim signature lifting.
