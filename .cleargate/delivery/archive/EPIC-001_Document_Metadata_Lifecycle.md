## 1. Problem & Value

**Why are we doing this?**
Without metadata, we cannot answer "when was this draft written? against what code state?" which makes CR blast-radius, stale detection, and cross-session continuity impossible. PROPOSAL-001 made the case; this Epic ships it.

**Success Metrics (North Star):**
- Every new document generated by a template has `created_at` + `created_at_version` set on first save.
- Re-stamping a file is idempotent (`created_at` never changes; only `updated_at` / `updated_at_version`).
- On `cleargate_push_item`, the MCP injects `pushed_at_version` into the response + frontmatter.
- Dirty-worktree drafts are visibly flagged (`a3f2e91-dirty`).

## 3. The Reality Check (Context)

| Constraint | Rule |
|---|---|
| Idempotency | `stamp <file>` must produce byte-identical output when called twice in a row with no code changes. |
| Dirty detection | `git status --porcelain` — non-empty output means dirty. |
| Archive rule | Once a file's path contains `/archive/`, `stamp` is a no-op (freezes). |
| Cross-platform | Must work on macOS and Linux at minimum. Windows out of scope for v1. |
| Token discipline | Frontmatter block adds ~5-6 lines per document. Small but cumulative; no optional fields. |
| CLI package dependency | `cleargate-cli/` does not yet exist. This Epic implicitly scaffolds it (or is blocked on a prerequisite scaffold story). |

## 5. Acceptance Criteria

```gherkin
Feature: Document metadata lifecycle

  Scenario: First stamp creates all fields
    Given a fresh file at .cleargate/delivery/pending-sync/EPIC-X.md without metadata fields
    When I run cleargate stamp on the file
    Then created_at, updated_at, created_at_version, updated_at_version are set
    And pushed_at_version is null

  Scenario: Re-stamp is idempotent for created_at
    Given a stamped file
    When I run cleargate stamp again with no changes to the file body or git state
    Then created_at is unchanged
    And updated_at advances (unless no content change either)

  Scenario: Dirty worktree produces -dirty suffix
    Given the git worktree has uncommitted changes
    When cleargate stamp runs
    Then updated_at_version ends with "-dirty"

  Scenario: Archive freezes updated_at
    Given a file located under .cleargate/delivery/archive/
    When cleargate stamp runs on it
    Then updated_at is unchanged

  Scenario: Git absent falls back to package.json
    Given a directory without .git but with package.json v1.4.2
    When cleargate stamp runs
    Then updated_at_version equals "1.4.2"
    And a warning is emitted

  Scenario: MCP push writes pushed_at_version
    Given an authenticated push_item call
    When the server persists the new version
    Then current_payload frontmatter has pushed_at_version = server-side codebase_version
```
