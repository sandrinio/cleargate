# Sales Memo — What AI-Driven SDLC Changes

**One-liner:** ClearGate collapses the gap between product planning and code delivery by putting the backlog *in the repo* and keeping it synced with the tools stakeholders already use.

## What changes with AI-Driven SDLC

### 1. Unified backlog, in the repo
Every team working in a codebase shares a single backlog that lives alongside the code — proposals, epics, stories, CRs, bugs — all as markdown under `.cleargate/delivery/`. No more context-switching between Jira, Linear, Notion, and the repo to understand what's being built and why. The agents see the same backlog the humans do.

### 2. Synced markdown for stakeholders
Non-engineering stakeholders (PM, design, exec, legal) write proposals, prioritize backlog items, and watch the full delivery process live — in the same markdown files the agents execute against. Two-way sync with the PM tool (MCP adapter, no middleman DB) means stakeholders keep their native tooling while engineering keeps the repo as source of truth.

## Why this matters (the pitch)

- **Before:** PM writes a ticket in Jira → engineer re-reads it in their IDE → AI assistant has no awareness of it → context is reconstructed every session → drift between plan and code is normal.
- **After:** The plan, the code, and the AI agent all read from the same files. Stakeholders see delivery progress in real time without asking. Agents inherit the full plan context on every invocation.

The change isn't "we added AI to the SDLC" — it's **the backlog moved into the repo**, and the SDLC became coherent for the first time.
